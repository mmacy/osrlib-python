# Phase 2 plan — combat kernel

Implementation plan for Phase 2 of [the osrlib spec](spec.md). Phase 2 delivers the monster compiler (variant stat blocks, category and immunity tags), the combat tables as data (attack matrix, monster saves, XP awards), and the combat kernel: attack resolution with weapon-material and graded-immunity damage, range bands, the shared targeting model, initiative, saving throws, morale, damage/healing/death, energy drain, and the effects engine with the core condition set. It also ships the first real events, the default English message formatter, and the event type-discriminator decision Phase 0 deferred. Milestone from the spec: deterministic scripted battles — a party versus 1d8 trolls with regeneration ticking, and a wight fight that drains a level — resolved entirely through kernel calls with full event streams.

## Scope

In scope:

- `tools/srd_compile/monsters.py` and `tools/srd_compile/combat_tables.py` — the monster and combat-table compilers, with `overrides/monsters.json` for parse corrections and hand tags
- `src/osrlib/data/` — generated `monsters.json` and `combat_tables.json` with typed loaders `load_monsters()` and `load_combat_tables()`
- `core/monsters.py` — frozen `MonsterTemplate` (stats, attack routines, defenses, ability tags, categories), mutable `MonsterInstance`, HP rolling and spawning
- `core/tables.py` — attack matrix lookup, monster saving-throw bands, the XP-awards-for-defeated-monsters table
- `core/effects.py` — the effect lifecycle engine (durations, periodic ticks, expiry, stacking, attachment to creatures/items/locations) and named conditions with combat hooks
- `core/combat.py` — attack rolls (matrix and THAC0-arithmetic variant), damage with material/immunity resolution, range bands, weapon-quality execution, initiative (side and individual), morale checks, saving throws, healing and death, equipment destruction on destructive death, falling damage, unarmed and splash attacks, and the shared targeting model
- `core/classes.py` grows energy drain as the inverse of leveling; `core/character.py` and `core/items.py` grow the additive fields and validators combat needs (conditions, derived THAC0/saves, two-handed-versus-shield equip conflict)
- `core/events.py` grows the first kernel event types plus the serialized type discriminator; `osrlib/messages.py` is created with the default English message formatter
- `core/ruleset.py` grows the five flags Phase 2 reads: `variable_weapon_damage`, `individual_initiative`, `thac0_arithmetic`, `weapon_reload`, `hd5_counts_as_magical`
- Tests: data pipeline and table fidelity, combat/effects/drain unit suites, property tests, and the two milestone golden battles

Out of scope (deferred to the phase that consumes them):

- The battle state machine, encounter sides, the per-group range track and combat space, ranks/formation, and the default monster action policy (4) — Phase 2's milestone battles are scripted sequences of kernel calls, not a combat loop
- Geometry-to-target mapping for areas (4) — Phase 2's targeting model resolves area and breath effects against explicitly supplied target lists, per the spec's Phase 3 milestone note
- Encounter procedure, surprise, reaction rolls (the 2d6 CHA-modified table), evasion/pursuit/exhaustion (4) — reaction rides with its consumer, the encounter procedure
- Fighting withdrawal (4) — its entire content is movement (half encounter speed, clear space behind), which belongs to the range track; retreat lands now because its content is attack modifiers
- Spell memorization, casting, disruption, and turning undead (3) — the combat-sequence "declare spells" step and disruption trigger are noted as seams; monster data ships the category tags (`undead`, `person`) that *charm*/*hold*/*sleep* and turning consume
- Thief percentile skill checks and demihuman detection checks (4) — back-stab's to-hit bonus and damage doubling land now via the Phase 1 class-ability tags; the d% check procedure lands with its exploration consumers
- Magic items and the `magic_item_death_save` flag (5) — the destructive-death procedure ships now and destroys mundane equipment; the magic-item saving throw needs magic items to exist, and shipping the flag before the behavior would be a lie in the API (Phase 1 precedent)
- Treasure generation (5) — monster treasure-type letters ship as faithful references now
- Encounter sizing from number-appearing values and the encounter tables (4) — NA dice ship as data now
- Sessions, commands, views, persistence (4) — Phase 2 kernel functions return result models carrying their events; entity IDs come from a lightweight allocator the Phase 4 session adopts
- Subduing, bombing, unstable surfaces, underwater combat, and cover (out of 1.0) — none appear in the spec's rules scope; cover-like situations are served by the caller-supplied situational attack modifier, which is the RAW shape (referee applies −1 to −4)
- XP award division and timing (5); statistical chi-square tests (5, they target treasure/reaction/wandering generators)

## Work items

### 1. Monster data model — `core/monsters.py`

- `MonsterTemplate` (frozen, `overrides_applied` like every compiled model): `id`, `name`, `page` (source-page grouping so variants stay associable), intro prose, `ac`/`ac_ascending` (optional ints), `attack_roll_required: bool` (False for the `No hit roll required` stat, see hazards), `hit_dice: MonsterHitDice`, `attacks: tuple[AttackRoutine, ...]` (alternatives — a monster acts with one routine per round), `thac0`/`attack_bonus`, `movement: tuple[MovementMode, ...]`, `saves: MonsterSaves`, `morale: int | None` plus `morale_alternates` (score + condition prose: `10 (8 fear of fire)`), `alignment: AlignmentSpec`, `xp: int` plus structured `xp_notes` (leader/chieftain/guard values), `number_appearing: NumberAppearing` (dungeon and lair/wilderness dice, zero and `see below` semantics preserved), `treasure: TreasureRef` (primary letters, parenthetical letters, `+ N gp`, `× N`, special labels like `Tusks` — stored faithfully, semantics pinned in Phase 5), `abilities: tuple[MonsterAbility, ...]` (tag + name + prose + `manual` + params, mirroring `ClassAbility`), `defenses: Defenses`, `categories: tuple[str, ...]`.
- `MonsterHitDice`: `count`, `die` (8 unless fractional — `½` compiles as 1d4), `modifier` (signed: `1-1` is −1), `asterisks`, `average_hp`, `fixed_hp: int | None` (`1hp` forms roll nothing). The asterisk count is XP data, not noise; the modifier drives the attack-matrix "1 HD higher" rule and the *sleep*-style fixed-bonus exclusions later.
- `Defenses` — the structured shape combat checks at damage time, compiled from ability bullets plus overrides while the bullets keep the prose: `harmed_only_by: tuple[DamageKey, ...]` (empty = no gate; wight is `(silver, magic)`, spectre `(magic,)`, mummy `(fire, magic)`, black pudding `(fire,)`), `reductions: tuple[DamageReduction, ...]` (wraith: half from silver; mummy: half from everything that passes the gate), `energy: dict[Element, EnergyDefense]` where `EnergyDefense` is `immune` or `auto_save` (dragons: immune to own element, auto-save versus similar forms), `condition_immunities` (undead poison/mind immunity, ghoul-paralysis carve-outs). `holy` is a `DamageKey` carried by holy water's combat facet, and pinned: an undead target admits holy damage through any `harmed_only_by` gate — the specific rule ("holy water inflicts damage on undead monsters") overrides the general immunity, otherwise the wight's silver-or-magic gate would absorb the one weapon made for it.
- Mechanized ability tags Phase 2 consumes, with params pinned by the compiler: `regeneration` (troll: `delay_rounds=3`, `per_round=3`, `blocked_by=(fire, acid)`, `revive="2d6"`; vampire: `delay_rounds=0`, `while_alive=True`), `energy_drain` (`levels`, `xp_policy` — see work item 10), `poison` (outcome death, optional onset delay, or damage riders), `paralysis` (duration dice, immunity carve-outs), `petrification` (vector: touch/gaze/breath), `breath_weapon` (targeting: an area shape with dimensions *or* single-target — the hellhound's breath is "One target"; element; `uses_per_day: int | None` — the three-per-day limit is the *dragons'* rule, pinned per-monster, `None` elsewhere; `per_round_chance` where the SRD gates use by chance — the hellhound's "2-in-6 chance per round" ships as data for Phase 4's action policy, the kernel resolves the breath whenever invoked; damage model `current_hp`, dice, or dice-per-HD; save-for-half or save-or-die), `gaze`, `disease` (mummy rot), `uses_fire` (burning-oil immunity marker). Everything else (charm details, swallow whole, rusting, blood drain, strength drain, shape changing, ...) compiles as a tag with `manual=True` plus prose — games and narrators can present it; the kernel doesn't execute it in Phase 2.
- `MonsterInstance` (mutable, spawned from the frozen template): `id`, `template`, `max_hp`/`current_hp`, `conditions`, damage ledgers for regeneration (see work item 6). Exposes the same combatant surface as `Character` (THAC0, attack bonus, AC both ways, saves) so combat functions take either.
- `spawn_monster(template, *, id, stream) -> MonsterInstance`: HP = sum of `count` d8 rolls (or the fractional/fixed form) + modifier, minimum 1 — pinned. Draws from the `"monster_spawn"` stream (module constant, following the `character.py` precedent) so a combat-rules change never shifts spawned HP in a golden.
- `IdAllocator` — tiny monotonic per-prefix counter (`monster-0001`, `effect-0001`) matching the spec's session-scoped ID contract; standalone users and tests hold one, the Phase 4 session adopts it.

### 2. SRD compiler — `tools/srd_compile/monsters.py` and `combat_tables.py`

Same shape as the Phase 1 parsers: `compile_monsters(srd_dir)` and `compile_combat_tables(srd_dir)` slot into `__main__.py` beside the existing four, validate through the shipped osrlib models, and write with the established deterministic conventions (sorted keys, sorted ids, `_meta` source pages).

- Page inventory: the 138 monster pages are exactly the entries of `srd/Monster_Descriptions.md` (the A–Z index), which the compiler reads as its manifest rather than guessing from filenames — `Game_Statistics_(Monsters).md` and `General.md` match a naive stat-block grep but are rules pages. The index maps Dwarf/Elf/Halfling/Invisible Stalker to their `_(Monster)` pages.
- Stat blocks are uniform 2-column key/value pipe tables with 11 rows in fixed order (Armour Class, Hit Dice, Attacks, THAC0, Movement, Saving Throws, Morale, Alignment, XP, Number Appearing, Treasure Type — no page deviates or reorders), fenced by horizontal rules, followed by `- **Label:** prose` ability bullets. Multi-variant pages carry one `##` section per variant, each with its own table and bullets, and shared abilities bulleted above the page's `## Contents`; shared bullets merge into every variant (per-variant `See main entry.` bullets resolve against them). IDs are the slugified variant heading (`red_dragon`, `fire_beetle`); `page` records the grouping.
- Packed-variant expansion, pinned: pages whose *cells* pack several creatures expand into one concrete entry per variant, because a frozen template must be spawnable (one HD value to roll). `Elemental.md` (slash-packed `2 [17] / 0 [19] / -2 [21]` across Lesser/Intermediate/Greater power levels) → `air_elemental_lesser` / `_intermediate` / `_greater` and likewise per element; `Vampire.md` (`7 to 9**`) → `vampire_7` / `vampire_8` / `vampire_9`; `Hellhound.md` (`3 to 7*`) → `hellhound_3` … `hellhound_7`; `Hydra.md` (`5 to 12`, `8hp per HD`, per-HD THAC0 and XP) → `hydra_5` … `hydra_12` with fixed HP = 8 × HD; `Insect_Swarm.md` (`2 to 4`) → `insect_swarm_2` … `insect_swarm_4` (its AC/morale/NA are single values shared by all three — cardinality is per-row, not per-block). The small herd animal (`1 to 2`) and the veteran (`1 to 3`) expand the same way. Where an expanded axis needs values the SRD prints once (the veteran's single save line spans F1–F3), the printed values ship on every expansion — checked against the fighter table, whose saves are constant across levels 1–3, so nothing is misstated.
- Parse hazards found during survey, recorded so the implementer doesn't rediscover them:
    - Dual formats everywhere: AC `4 [15]`, THAC0 `13 [+6]` (also `19 [0]` and `20 [-1]`), both values parsed and shipped, reusing the Phase 1 regexes.
    - Nested brackets: lycanthrope AC `3 [16] (9 [10] in human form)` — the parenthetical alternate form compiles as a structured alternate, not a parse error.
    - Attack routines: `1 × mace (1d6)`; multi-attack `2 × claw (1d3), 1 × bite (1d6)`; bracket-grouped alternatives `[2 × claw (1d4 + 1), 1 × bite (2d10)] or breath` (dragons — damage dice contain spaces); labeled sub-attacks `2 × claw (1d3), goat: 1 × gore (2d4), lion: 1 × bite (2d4), dragon: 1 × bite (3d4) or 1 × breath (3d6)` (chimera — labels fold into attack names); `1 × weapon (1d6 or by weapon)` and modified `(1d4 or by weapon -1)` (by-weapon flag + modifier); effect keywords in the damage parens (`+ poison`, `+ paralysis`, `energy drain`, `(charm)` ...) compile to effect tags on the attack; fixed damage `1 × bite (1hp + poison)`; the insect swarm's `1 × swarm (2 or 4hp)` (fixed alternatives — armour-dependent per its prose, which stays `manual`); bone golem's `2 or 4 × weapon` normalizes to two alternative routines via override.
    - Movement: multi-mode `90' (30') / 240' (80') flying` with descriptor words (`flying`, `swimming`, `gliding`, `in human form`, `in webs`); typographic `'` (U+2019) feet marks and `×` (U+00D7) multipliers throughout.
    - Saving throws: `D12 W13 P14 B15 S16 (2)` — the parenthetical is save-as (HD number, `NH`, a class form like `(Cleric 1)` / `(Thief 1)` / `(F1 to F3)`, or absent: hydra's `By HD`, elemental's `See main entry (8 / 12 / 16)` resolve during expansion, hydra's from the monster save bands).
    - Morale conditional forms are common: `8 (10 with leader)`, `9 (12 in lair)`, `10 (8 fear of fire)` → base + alternates; `Varies` (merchant) → `None`.
    - Non-numeric sentinels: AC `No hit roll required` (green slime, yellow mould) → `attack_roll_required=False`; NA `0 (see below)` (buccaneer, pirate); Treasure `Tusks` / `Horn` / `Honey`, `E + 5,000gp`, `V × 3`, `U (A)`.
    - Data typos to fix via overrides with reasons: gnome XP `10, (leader: 20, bodyguard: 35, chieftain: 75),` (stray commas), centaur attacks trailing comma, the lycanthrope grammar slip `can only harmed by`.
    - Markdown noise inside cells: bold `**HD**` (hydra, hellhound), en-dashes in prose ranges, double-space bullets (`-  **Surprise:**`), `½` (U+00BD), double-URL-encoded filenames (`Doppelg%25C3%25A4nger.md`).
    - HD forms: `6+3*`, `1-1 (3hp)`, `½* (2hp)`, `1hp`, `9**` — single and double asterisks are the special-ability count for XP; `(8hp per HD)` on hydra.
- Category tags, pinned derivations:
    - `undead` — auto-tagged from the standard verbatim bullet (`**Undead:** Make no noise, until they attack. ...`) present on exactly ghoul, mummy, skeleton, spectre, vampire, wight, wraith, zombie. The shadow explicitly is *not* undead ("incorporeal (but not undead)") and gets no tag — the census makes silent mis-tagging detectable by count.
    - `person` — parsed from the authoritative suggested list in `General.md` §Persons (33 entries, acolyte through veteran). The rule text delegates to referee judgment, so the SRD's own list is the pinned default; games can diverge via their own data.
    - `enchanted` — no parseable marker exists in the SRD text; hand-curated in `overrides/monsters.json` with a reason per entry (gargoyles, golems, elementals, living statues, and kin). Pinned as the register's documented mechanism for category judgment calls.
- Immunity bullets are keyword-matched, not exact-string-matched — the census of forms: `Mundane damage immunity` (silver-or-magic, magic-only, lycanthrope animal-form conditional), `Damage immunity` (mummy fire-or-magic + half), `Damage reduction` (wraith half-from-silver), `Immunity`/`Fire immunity`/`Energy immunity`/`Spell immunity` variants. Every compiled `Defenses` shape is override-correctable and the fidelity tests spot-check the tricky ones (wight, wraith, mummy, spectre, black pudding, green slime, yellow mould, elemental, gargoyle).
- XP is pre-calculated per stat block and ships as data — no lookup needed at runtime. The compiler cross-validates each entry against the `Awarding_XP.md` table: base by HD row (including the separate `N+` rows), plus asterisk count × the bonus column, where above 21 HD *both* the base and the bonus amounts first gain 250 per HD above 21 ("add 250 XP to the Base and Bonus amounts") — the dragon turtle (HD 30*, XP 9,000) proves the reading: (2,500 + 9×250) + 1 × (2,000 + 9×250) = 9,000. Pinned: negative hit-point modifiers map to the *lower* band — the goblin's 1-1 HD validates against the "Less than 1" row (XP 5) and keeps THAC0 19 [0], the "Up to 1" matrix row; the "attack as 1 HD higher" rule is for bonus modifiers only. Mismatches fail the build unless recorded in the overrides file with a reason, so SRD typos surface once, at compile time.
- `compile_combat_tables` output: the attack matrix (16 THAC0 rows from `20 [-1]`/NH to `5 [+14]`, AC columns −3..9, with the monster-HD row labels), the monster saving-throw bands (NH, 1–3, 4–6, 7–9, 10–12, 13–15, 16–18, 19–21, 22+), and the XP-awards table. Source pages: `Combat_Tables.md`, `Awarding_XP.md`.

### 3. Generated data, loaders — `src/osrlib/data/`

- `monsters.json` and `combat_tables.json`, each with the `_meta` block; the monster `_meta.source_pages` lists all 138 pages plus `General.md` (person list) and `Monster_Descriptions.md` (manifest). Never hand-edited; the existing CI regeneration job covers the new outputs automatically since it diffs the whole data directory.
- Loaders `load_monsters() -> MonsterCatalog` and `load_combat_tables() -> CombatTables` in `osrlib/data/__init__.py`, following the exact Phase 1 pattern (`importlib.resources`, `@cache`, frozen models, `ContentValidationError` on validation failure, id-uniqueness validator, `.get(id)` raising `ValueError`).

### 4. Ruleset — `core/ruleset.py`

Five new flags, defaults per the spec, read at resolution time:

- `variable_weapon_damage` (default on) — off means every weapon *and gear combat facet* deals 1d6 (RAW: "PC attacks inflict 1d6 damage"); unarmed attacks stay 1d2 (the specific unarmed rule, not a weapon) and monster damage is unaffected (monsters always "deal the damage indicated in the description") — pinned.
- `individual_initiative` (default off) — 1d6 per participant, DEX-modified for characters (plus the halfling's `initiative_bonus` tag); monsters take a caller-supplied modifier (default 0), the RAW referee-judgment surface.
- `thac0_arithmetic` (default off) — replaces the matrix lookup with unclamped `THAC0 − AC` subtraction. The ascending-AC attack procedure (roll + attack bonus ≥ AAC) is algebraically identical to THAC0 subtraction, so this one flag covers both presentations; the matrix differs only through its clamping (see work item 7).
- `weapon_reload` (default off) — a reload-quality weapon may not fire two rounds running: the attack validator rejects when the caller-supplied context says the weapon fired last round. Round bookkeeping is the Phase 4 battle machine's job; the kernel enforces the rule given honest context — pinned.
- `hd5_counts_as_magical` (default off) — implements the SRD invulnerabilities optional rule *in full*: both a monster of 5+ HD and another invulnerable monster bypass silver/magic-only gates. The spec's flag summary names the 5+ HD half as shorthand; the flag governs the whole optional rule it cites, since implementing half an optional rule would be an undocumented deviation — pinned. Boundary pinned to the rule's own wording ("monsters [that] can only be harmed by magical or silver weapons"): the flag touches only weapon-material gates whose keys are a subset of {silver, magic}, and "another invulnerable monster" means a monster with such a gate — element-keyed and mixed gates (mummy's fire-or-magic, black pudding's fire-only, green slime's cold-or-fire) are unaffected.

### 5. Events and the message formatter — `core/events.py`, `osrlib/messages.py`

- The Phase 0 deferred decision, now due — serialized type discriminator, pinned: every event class declares `event_type: Literal["attack_rolled"]`-style wire field (snake_case, schema-stable, additive-only). Rationale: pydantic discriminated unions on a literal field give native tagged-union JSON Schema for the FastAPI/LLM consumers and mechanical "ignore unknown event types" (skip unknown discriminator values); message `code` stays free to be outcome-bearing without touching the union.
- Codes are outcome-bearing, pinned: one event class may emit several codes from a declared closed set (`combat.attack.hit` / `combat.attack.missed` / `combat.attack.no_effect` on the attack event), so the formatter and narration key off codes while consumers discriminate on `event_type`.
- First kernel event set (all frozen `Event` subclasses in `core/events.py`, structured fields + entity IDs only): initiative rolled, attack resolved (with per-die rolls, modifiers, hit AC), damage dealt (amount only — never the target's remaining HP: monster HP is hidden by design; a referee-visibility state event carries it), immunity absorbed the hit, saving throw resolved, morale checked (referee visibility — players learn the outcome from behavior, not the roll), condition gained/removed, effect attached/ticked/expired/released, healing applied, death, equipment destroyed, level drained, monster revived (troll). Every event declares its visibility deliberately; the referee sees everything.
- `osrlib/messages.py` — the default English formatter the spec promises alongside the first real events: `format_message(event) -> str`, pure string templating keyed by `code`, no I/O. Total function, pinned: unknown codes format to the code string itself rather than raising, honoring "consumers must ignore unknown event types" for forward compatibility. A test asserts every shipped code has a real template.
- Kernel emission shape, pinned: combat and effects functions return frozen result models carrying an `events: tuple[Event, ...]` field. This satisfies the milestone ("full event streams" from kernel calls, no session) and the spec's return-based emission rule; the Phase 4 session will append `result.events` to its log. The hot-path concern stays satisfied — à la carte callers read the plain result fields and ignore events; `model_construct` remains the sanctioned fast lane if profiling ever demands it.

### 6. Effects engine and conditions — `core/effects.py`

- `EffectDefinition` (frozen): duration (`rounds`/`turns`/`days` count, or indefinite, or permanent), periodic tick spec (interval in rounds + tick behavior), stacking rule (`stack`, `refresh`, `ignore`), expiry outcome, and the condition it grants, if any. `ActiveEffect` (mutable): `effect_id`, definition, target ref (creature, item, or location id — a burning oil pool attaches to a location string now; cells arrive in Phase 4), attachment round, bookkeeping state (remaining ticks, accumulated counters).
- `EffectsLedger` — the serializable engine: `attach(...)`, `release(effect_id)`, and `advance(clock, n, unit, registry) -> list[Event]`, which advances the `GameClock` and resolves each round boundary internally (the clock reports turn/day crossings; per-round ticks are the ledger's own iteration). Canonical tick order per the spec, locked by test: at each boundary, expirations resolve before ticks; simultaneous effects resolve in attachment order, tie-broken by effect id. `registry` maps entity ids to live objects (the Phase 4 session will own one; tests pass a dict).
- Conditions: a `Condition` str enum shipping the spec's full named set (paralysed, asleep, blind, charmed, petrified, diseased, exhausted, lycanthropy_incubation, averted_eyes, poisoned, dead) — the enum is cheap and additive-safe; Phase 2 implements combat hooks for the set it consumes: paralysed/helpless (auto-hit in melee, damage roll only), asleep (helpless + dies to a blade — the condition hook ships now, the *sleep* spell that inflicts it is Phase 3), blind (cannot attack), averted_eyes (−4 to hit; the gazer gains +2 against the averter), petrified (cannot act; recoverable — stone is not dead), poisoned/diseased as marker states driving their effects.
- Single-writer rule, pinned: `Character` and `MonsterInstance` carry `conditions: tuple[ActiveCondition, ...]` (condition + owning effect id) so a serialized creature is honest on its own, but only the effects engine writes it — combat reads conditions locally, and an invariant test asserts ledger and creature state never desync.
- Petrification interplay, pinned: while a target is petrified its other attached effects suspend (no ticks, durations frozen) — a poisoned, petrified adventurer is a problem for after *stone to flesh*.
- Poison: save versus death/poison; failure is death (immediate, or an onset-delay effect for forms like the giant rattler's "death in 1d6 turns" and the medusa's "die in 1 turn"); poison riders on damaging attacks leave the damage unaffected by the save, per RAW.
- Regeneration as data-driven effects: troll — after each instance of damage, a 3-round delay, then 3 hp per round; fire and acid damage accrue in a separate non-regenerable ledger on the instance; regeneration never heals that ledger, and pinned: the troll is permanently dead only when non-regenerable damage alone reaches max HP — otherwise death at ≤ 0 hp schedules a 2d6-round revival (the `"effects"` stream), emitting the revival event when it lands. Vampire — 3 hp at the start of each round while alive, no delay, no revival (its 0 hp gaseous escape stays `manual` prose).
- Mummy rot, pinned: while diseased, magical healing is ineffective and natural healing runs ten times slower — one 1d3 recovery per ten consecutive full rest days, tracked on the effect; removal is Phase 3 magic.
- Splash dousing (from work item 8): the listed damage applies on the hit and once more at the next round boundary, then expires — "inflicted for two rounds", pinned as two applications.
- Consumable timers (torch 6 turns, lantern 24) are expressible with this engine by construction — a test lights a torch-shaped effect and expires it — but the light-source *rules* land with Phase 4 exploration.
- Effect-internal randomness (revival delays, onset dice, natural-healing rolls) draws from the `"effects"` stream (module constant); battle resolution draws never shift effect draws and vice versa.

### 7. Combat tables — `core/tables.py`

- Attack matrix from `combat_tables.json`: `to_hit_ac(thac0, ac) -> int`. The shipped matrix is asserted verbatim against the SRD, and the survey's structural finding is locked as a property test: every printed cell equals `clamp(THAC0 − AC, 2, 20)`. Pinned: AC values outside the printed −3..9 columns extend by the same formula — the printed bounds are page layout, not a rules cliff. The clamping is exactly what distinguishes matrix mode from the `thac0_arithmetic` flag once modifiers push totals past the plateaus (a modified 20 hits matrix-required-20 but misses arithmetic-required-21; a modified 0 misses matrix-required-2 but hits arithmetic-required-≤1 on a natural 2+).
- Monster THAC0 comes printed in every stat block (already reflecting the "bonus hit points attack as 1 HD higher" rule — troll 6+3 carries THAC0 13); the HD→row mapping ships for validation and custom monsters.
- Monster saving throws: `monster_saves(hd_band) -> SavingThrows` over the nine bands including NH; monster stat blocks carry explicit values, so this table serves validation, custom monsters, and the save-as resolutions from expansion.
- XP awards table: `monster_xp(hit_dice, asterisks) -> int` including the `N+` rows, the shared 7–7+ style bands, and the above-21-HD inflation of both the base and bonus amounts (work item 2) — used by the compiler cross-check and available for custom monsters.

### 8. Combat procedures — `core/combat.py`

- Combatant surface: a `Combatant` protocol (id, THAC0/attack bonus, descending and ascending AC, `SavingThrows`, current/max HP, conditions, and the modifier hooks). `Character` grows derived `thac0`, `attack_bonus`, and `saves` properties reading `definition.row(level)` — never stored; `MonsterInstance` exposes the same names from its template. Kernel functions take either.
- `validate_attack(attacker, defender, attack, context) -> list[Rejection]` — the pure pre-phase, mirroring Phase 1's validator convention and Phase 4's command-rejection contract: a dead or petrified or paralysed attacker cannot act, a blind attacker cannot attack, melee beyond 5 feet, missile beyond long range, reload violations under the flag. No RNG draws, no mutation.
- `attack_roll(attacker, defender, attack, *, context, ruleset, stream) -> AttackRollResult` — 1d20 from the `"combat"` stream; modifiers: STR melee / DEX missile, range band (+1 short, −1 long, impossible beyond long), the caller-supplied situational modifier (cover, dragon-dozing +2, and kin — the RAW referee surface), back-stab (+4 to hit *and* doubled damage, both halves from the thief tag's `attack_bonus`/`damage_multiplier` params, when the context asserts an unaware target from behind), halfling `missile_attack_bonus`, retreat context (+2 against a retreating defender, shield AC ignored; shield also ignored when attacked from behind), the halfling `defensive_bonus` against large attackers (context flag). Natural 20 always hits, natural 1 always misses; helpless defenders are hit automatically in melee (no roll consumed, damage only, per RAW). Resolution is matrix lookup or THAC0 arithmetic per the flag; both AC presentations supported throughout.
- Damage pipeline, order pinned: (1) immunity gate — if the defender's `harmed_only_by`/energy immunities exclude this `DamageSource` (material, enchantment, element, `holy`, kind, destructive flag), no damage is rolled and the event says so; (2) damage roll (weapon dice or 1d6 under the flag; monster routines as compiled; unarmed 1d2) + STR for melee, then quality/context doublings (brace, charge, back-stab), minimum 1 on a hit; (3) reductions (wraith half-from-silver, mummy half-everything) floor but not below 1; (4) `apply_damage` mutates `current_hp` (floor 0), routes fire/acid into the troll's non-regenerable ledger, and emits death at 0 — "a character or monster reduced to 0 hit points or less is killed". Save-for-half halving also floors — pinned.
- Weapon qualities executed here: slow (acts last — ordering exposed via initiative results, see below), two-handed (cannot use a shield: equip-time rejection, work item 11), brace (double damage against a charging attacker when context says braced) and charge (double damage when the context asserts a 60'+ mounted charge — the kernel checks the quality and doubles; supplying the movement context is the caller's or Phase 4+'s job), reload (work item 4), splash.
- Splash weapons, pinned: a hit attaches the two-round dousing effect (work item 6). Holy water affects only the undead — against anything else the throw resolves normally but the damage pipeline reports no effect (the `holy` source key is only admitted by undead targets), symmetric with burning oil against `uses_fire` monsters and RAW-shaped: throwing holy water at a bandit is possible, just useless. Deliberately *not* a validator rejection — rejections are free (no roll, no time, no log entry per the spec), and a rejection would be a zero-cost undead detector, leaking exactly the kind of outcome B/X hides until it matters. Burning oil requires the lit-context flag; monsters tagged `uses_fire` are immune outright; unlit oil deals no damage and may be compiled into a location-attached pool effect (3' pool, burns 1 turn once lit, 1d8 to creatures passing through — who passes through is the caller's assertion until Phase 4 owns space).
- Initiative: `roll_side_initiative(side_keys, stream)` (1d6 per side, ordered high to low) and `roll_individual_initiative(participants, stream)` under the flag (1d6 + DEX modifier + class tag + caller monster modifier). Ties re-roll, pinned — RAW offers "re-roll or simultaneous", and simultaneous resolution is a different combat model the engine doesn't attempt; tied sides (or tied individuals, among themselves) re-roll in stable input order until distinct, each re-roll consuming draws. Slow-weapon actors are flagged in the result ordering: they act after all non-slow actors, ordered among themselves by their side's initiative (under individual initiative, by their own results) then stable order — pinned reading of "always act last, as if they had lost initiative" for the general case.
- Morale: `check_morale(score, *, modifier=0, stream) -> MoraleResult` — 2d6 versus ML; over means flee/surrender; ML 2 never fights, ML 12 never checks, and situational adjustments (clamped to ±2 per RAW) never apply to 2 or 12. A `MoraleTracker` carries the two-passed-checks memory ("fights until killed, no further checks"). Trigger detection ships as a queryable helper — first death on a side, half the side incapacitated, where incapacitated is pinned as dead or paralysed/petrified/asleep (RAW: "slain, paralysed, etc") — invoked by the milestone scripts now and the Phase 4 battle machine later. Conditional morale scores (troll 8 versus fire) are data; the caller picks which applies.
- Saving throws: `saving_throw(target, category, *, modifier=0, magical=False, stream) -> SaveResult` — 1d20 ≥ the value from the character's progression row or the monster's stat block. The WIS magic-save modifier applies to characters when `magical=True` and the category is not breath — pinned reading of "does not normally include saves against breath attacks" (referee discretion beyond that arrives as a caller modifier). Successful saves halve damaging effects (floored) and negate others; energy `auto_save` defenses treat the save as passed without a roll.
- Healing and death: `apply_healing(target, amount)` (instantaneous, capped at max HP), `natural_healing(target, stream)` — 1d3 for a full day of complete rest, callable by whoever can attest the rest was uninterrupted (Phase 4 automates); both consult condition hooks (mummy rot). Death at 0 emits the death event; equipment destruction on destructive death — when the killing `DamageSource` is flagged destructive (dragon breath now; *lightning bolt* in Phase 3) the victim's equipment is destroyed with an event; the magic-item save joins in Phase 5 with magic items.
- `falling_damage(feet, stream)` — 1d6 per full 10' fallen, floored, pinned.
- Energy drain application lives in `classes.py` (work item 10); `combat.py` wires the wight/wraith/spectre/vampire attack tags to it.

### 9. The targeting model — in `core/combat.py`

The shared shape spells, breath weapons, and thrown weapons resolve through, per the spec — Phase 2 defines the modes and resolves them against explicitly supplied candidate lists (geometry arrives with Phase 4's combat space):

- `TargetingMode` variants: `self`; `single`; `up_to_n` (fixed or rolled N — *hold person*'s 1d4 — with an optional per-mode save modifier such as its single-target −2); `hd_budget` (dice budget, pinned resolution: candidates sorted weakest-first by effective HD — sub-1 HD rounds up to 1, fixed hit-point bonuses dropped — consumed in order, ties broken by stable input order; the budget spends whole creatures, a target whose HD exceed the remainder is skipped and selection continues — *sleep*'s exact arithmetic lands with the spell in Phase 3, the resolver ships and is tested now); `area` (shape + dimensions: radius, line, cone, cloud — dragons' shapes compile with the page-level defaults where a variant omits dimensions); `gaze`.
- `select_targets(mode, candidates, stream) -> tuple[targets, events]`; each affected target then resolves its own save when the effect allows one, with half-on-save carried in the effect data.
- Gaze: `resolve_gaze(gazer, engaged, stream)` applies per-round exposure to each engaged combatant unless averting eyes (condition, with its ±modifiers) — mirror counterplay stays `manual` prose. Breath weapons resolve from their per-monster tag (work item 1): damage equal to current HP with save-for-half (dragons, three uses per day tracked on the instance), dice-per-HD single-target (hellhound — no daily limit; its 2-in-6 per-round gate is action-policy data for Phase 4), or save-or-die (sea dragon spittle).

### 10. Energy drain — `core/classes.py`

- `drain_levels(character, definition, *, levels=1, xp_policy, stream) -> DrainResult` — the inverse of `level_up`, as Phase 1 promised: saves, THAC0, and spell slots need no reversal because they derive from `row(level)`; only stored state reverses. Per level drained, mirroring `level_up` exactly in reverse: above name level subtract the flat-bonus delta (no roll, no CON); otherwise roll the class hit die + CON modifier (minimum 1 per die, the `"advancement"` stream — the same subsystem as the gains it reverses) and subtract it from max and current HP. Pinned: rolling the lost die is the RAW-faithful reading of "loses one Hit Die of hit points" that keeps the model stateless — no per-level HP history field.
- Pinned floors: drain never reduces max HP below 1 or current HP below 1 while the character retains a level — death by drain happens only by losing the last level ("a person drained of all levels" per the SRD), which is the terminal state: the character dies, and the spawn consequence (rises as a wight in 1d4 days, under control) is emitted as a structured field on the drain event with the prose `manual` — the kernel kills, the game narrates.
- XP after drain is per-monster data, pinned from the survey: the wight sets XP "halfway between the former and new levels" — floored midpoint of the two level thresholds; wraith, spectre, and vampire set XP "to the lowest amount for the new level" — the new level's threshold exactly. `xp_policy ∈ {halfway, level_minimum}` rides the monster's `energy_drain` tag; two-level drains (spectre, vampire) apply the procedure twice and then set XP once by policy.
- Draining a monster loses Hit Dice symmetrically (the SRD says "experience level (or Hit Die)"): the instance re-derives THAC0/saves from the reduced HD via the tables and loses a rolled d8; a monster drained below 1 HD dies — pinned.
