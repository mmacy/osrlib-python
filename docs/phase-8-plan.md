# Phase 8 plan — release

Implementation plan for Phase 8 of [the osrlib spec](spec.md). Phase 8 releases what Phase 7 documented: version 1.0.0, the changelog, the completed pyproject metadata (including the `Documentation` URL the Phase 7 site created), the packaging audits the Phase 6 plan named — the wheel-content audit and the clean-environment install smoke — the tag-driven trusted-publishing workflow, publication to PyPI as `osrlib` (the name confirmed still unregistered at planning time — the JSON API and the simple index both 404), and the install-instruction swap from source to PyPI that every README and docs page has honestly deferred since the Phase 7 rewrite. It also discharges the two decisions earlier phases explicitly parked here: the one test consequence of the version bump (the engine-version stamp normalization in the Phase 4 and Phase 5 golden comparisons, named in the Phase 6 plan) and the versioned-documentation decision (named in the Phase 7 plan). The milestone: `pip install osrlib` works in a clean environment anywhere, and every instruction that says so is true.

Phase 8 is a zero-behavior phase with an empty `src/` diff. The version moving to 1.0.0 is metadata, not behavior: the golden files stand byte-for-byte and are never regenerated for a version bump — the two party-document comparisons learn to normalize the stamp instead, the sanctioned fix the Phase 6 plan reserved for this phase — `SCHEMA_VERSION` stays 2 with no new migration, no serialized name moves, and `docs/adaptations.md` gains nothing, because Phase 8 pins release engineering, not rules. Everything this phase adds lives outside the package: the changelog, the metadata, two maintainer scripts, one workflow, and prose.

## Scope

In scope:

- The version bump to 1.0.0: pyproject, the lockfile refresh, the two golden-test stamp normalizations, and the one stale sample string in the FastAPI example README
- `CHANGELOG.md`: the 1.0.0 entry and the accrual convention for everything after it
- pyproject metadata completion: the license expression and license-files, classifiers, project URLs, and keywords
- The packaging audits as permanent maintainer tooling: the wheel/sdist content audit and the clean-environment install smoke, runnable locally and run on every release
- The release workflow: tag-driven, re-running the full standing gate, building once, auditing and smoke-testing the artifact on both OSes, publishing via PyPI trusted publishing, and creating the GitHub Release from the changelog
- The install-instruction swap: README, the docs index, and the quickstart move from "install from source" to PyPI instructions
- The versioned-documentation decision Phase 7 deferred here
- The releasing runbook in `AGENTS.md` and the post-merge release checklist in this plan

Out of scope (deferred to the phase or track that picks each up):

- Versioned-documentation adoption (mike or equivalent) — the decision is made in this phase (work item 6): not at 1.0. The adoption's pickup is named there: the first post-1.0 release whose published docs must describe behavior different from `main` adopts it in that release's own plan
- A PR-time `uv build` + audit CI job — deferred: the release workflow re-audits every artifact that can actually reach PyPI, and a per-PR build gate gates nothing a PR can break today. Pickup: reconsider if a release run ever fails on a packaging regression a PR gate would have caught
- A TestPyPI rehearsal — rejected, pinned: the content audit and the clean-environment smoke exercise the exact artifact that publishes; TestPyPI would validate upload plumbing that trusted publishing already standardizes, at the cost of a second pending publisher and a parallel project to maintain
- A `__version__` symbol or any runtime version re-export — rejected, pinned: the surface is frozen, the package root deliberately re-exports nothing, and `osrlib.versioning.engine_version()` reading installed metadata is already the one home for the running version
- Publishing the maintainer's email in package metadata — the owner's call, not the plan's: metadata is immutable once published. The default is name-only `authors`, standing unless the owner says otherwise at review
- Shipping the changelog in the sdist and a docs-site changelog page — rejected, pinned in work item 2
- Dependabot, action SHA-pinning, and workflow-hygiene tooling — post-1.0 maintenance; nothing in this phase makes them load-bearing
- New content of any kind — Advanced Fantasy content, the LLM-referee example, wilderness and the other spec-named exclusions — the standing post-1.0 tracks

## Work items

### 1. Version 1.0.0 and the stamp normalization — `pyproject.toml`, `uv.lock`, `tests/test_phase4_goldens.py`, `tests/test_phase5_goldens.py`, `examples/fastapi_crawler/README.md`

- The bump, pinned: `version = "1.0.0"` stays a static string in `pyproject.toml` — one source of truth, no dynamic-versioning machinery, because `engine_version()` reads installed metadata at runtime and nothing else needs the string. `uv lock` refreshes the lockfile in the same commit, and the lockfile diff must touch only the `osrlib` entry — no opportunistic dependency bumps ride the release commit. CI's `uv sync --locked` is what makes the refresh non-optional.
- The one test consequence, pinned: exactly two tests change — `test_party_rebuilds_from_the_creation_streams` in `tests/test_phase4_goldens.py` and `test_party_document_matches` in `tests/test_phase5_goldens.py` — and they change by adopting the normalization `tests/test_creation_goldens.py` already performs, stripping `engine_version` before the canonical comparison, with two local adjustments: the strip happens on copies, not the originals (the next bullet — the creation-goldens test pops test-local dicts and can afford to), and a companion assertion that the live document's stamp equals `engine_version()` rides beside each comparison (the creation suite keeps that assert in its own test), so the stamp stays tested without welding the goldens to a version string.
- The copies are load-bearing, pinned: both files' `golden` fixtures are module-scoped, and the Phase 4 `replayed` fixture feeds `golden["party_document"]` straight into `replay_game` — popping the key from the shared dict would pass today (the loader does not require the stamp) while planting test-order coupling for whoever adds the next assertion. Never mutate the fixture; compare stripped copies.
- The goldens themselves, pinned: they keep their `"0.1.0"` stamps forever. The stamp records the engine that produced the artifact, the loader does not gate on it, and replay determinism is untouched because no rules changed. Regenerating a golden for a version bump is a defect by definition — that sentence goes in the `AGENTS.md` runbook (work item 6) because it is exactly the "fix" a future contributor would reach for.
- The census, executed at planning time: outside this work item's files, the only `0.1.0` occurrences in the repository are the three golden stamps (which stay) and historical statements in the Phase 6 and Phase 7 plan documents (design documents, excluded from the site, untouched). The FastAPI example README's sample handshake — `"engine_version": "0.1.0"` in its curl transcript — updates to `1.0.0`. Nothing in `src/` states a version anywhere.

### 2. The changelog — `CHANGELOG.md`

- The format, pinned: [Keep a Changelog](https://keepachangelog.com/) at the repository root, with the header's semantic-versioning statement — at 1.0.0 that statement becomes the project's public versioning promise, and `schema_version` remains the separate, integer-valued serialization axis the spec already defines.
- The 1.0.0 entry, pinned: a single `Added` section bulleted by capability, not by phase — the determinism contract; characters, classes, equipment, and leveling; the combat kernel; magic; the crawl and the battle state machine; the `GameSession` command/event API with views; treasure, magic items, and NPC parties; save/load/replay with stamped documents and migrations; the two example front ends; the documentation site; the typed surface. Phases are development artifacts, not release deltas — the phase plans and git history are their archive, the same reasoning that killed the README's phase ledger in Phase 7.
- The accrual convention, pinned: an `[Unreleased]` section heads the file from day one; a PR that changes user-visible behavior adds its bullet there in the same PR, and a release renames the section to the version and date. The convention is written into the `AGENTS.md` runbook (work item 6) so it survives this plan.
- Distribution, pinned: the changelog does not ship in the sdist — the sdist is a build input, not an archive — and gets no docs-site page or README section while it has one entry and no external consumers. Discovery is the `Changelog` entry in `[project.urls]` (rendered in the PyPI sidebar) and the GitHub Release notes, which are the tagged version's section verbatim (work item 4).

### 3. Metadata completion — `pyproject.toml`

- The license expression, pinned: `license = "MIT AND LicenseRef-OGL-1.0a"` with `license-files = ["LICENSE", "LICENSE-OGL.md"]`, so both texts land in the wheel's `dist-info/licenses/`. The wheel factually ships Open Game Content — `osrlib/data/*.json` and its own `LICENSE-OGL.md` copy — and PEP 639's `LicenseRef-` form exists precisely for licenses off the SPDX list; a plain `MIT` expression would tell SBOM and license-audit tooling a false story and hide the Section 15 preservation obligation the spec makes binding. The fallback, pinned now so implementation doesn't improvise: if the uv_build backend or the PyPI upload rejects the `LicenseRef` expression, the expression falls back to `MIT` with both files still in `license-files`, this plan amends with the observed error, and the split remains stated by the data package's own OGL copy, the README, and the site licensing page. Either way, the content audit (work item 4) asserts both license texts ship, so the obligation is enforced independently of the expression.
- The classifiers, pinned as the exact list — and no `License ::` classifier beside an SPDX expression, per PEP 639: `Development Status :: 5 - Production/Stable`, `Intended Audience :: Developers`, `Operating System :: OS Independent`, `Programming Language :: Python :: 3`, `Programming Language :: Python :: 3.14`, `Topic :: Games/Entertainment :: Role-Playing`, `Topic :: Software Development :: Libraries`, `Typing :: Typed`. Only 3.14 is claimed because only 3.14 is tested; `requires-python = ">=3.14"` remains the actual constraint, and future minors earn their classifier by entering CI.
- `[project.urls]`, pinned: `Documentation` (the Phase 7 site — the URL that phase's plan promised this phase would record), `Repository`, `Issues`, and `Changelog`. `Homepage` is deliberately absent: `Documentation` and `Repository` cover both candidate targets, and a duplicate label is sidebar noise.
- Keywords: a short lowercase list (`bx`, `dungeon-crawler`, `game-engine`, `old-school-essentials`, `osr`, `rpg`, `srd`, `ttrpg`); implementation may trim, the pin is that the list exists.
- What does not change, pinned: `authors` stays name-only (the owner decision named in the scope), the dependency groups stay PEP 735 groups rather than published extras — dev, docs, and examples dependencies are contributor tooling, not user-installable features — and `dependencies = ["pydantic>=2"]` stands as the whole runtime story.

### 4. The packaging audits and the release workflow — `tools/release/check_dist.py`, `tools/release/install_smoke.py`, `.github/workflows/release.yml`

- Permanent over one-time, pinned: both audits are checked-in, stdlib-only scripts under `tools/release/` — beside `tools/srd_compile/` as maintainer tooling, never shipped, and the audit itself proves that — run as jobs in the release workflow so every future tag re-proves the artifact, and runnable locally so the first tag is de-risked before it exists. A one-time manual audit rots the day a data file moves.
- `check_dist.py`, the content audit: given a dist directory and an expected version, it asserts the wheel is tagged `py3-none-any` and contains the `osrlib/` module tree, `py.typed`, every `osrlib/data/*.json` with the member count equal to the checked-in `src/osrlib/data` listing (a dropped data file fails), the data package's `LICENSE-OGL.md`, and both license texts under `dist-info/licenses/`; that the wheel `METADATA` carries the expected name, version, license expression, and `Requires-Python`, and no `License ::` classifier; that the sdist carries `pyproject.toml`, the README, both license texts, and the full `src/osrlib/` tree; and that neither artifact contains any member from `tests/`, `docs/`, `srd/`, `site/`, `tools/`, `examples/`, or any `__pycache__`/`.pyc`. If uv_build turns out not to copy the root `LICENSE-OGL.md` into the sdist, the fix is the backend's include configuration and a plan amendment — surfacing exactly that is the audit's job.
- `install_smoke.py`, the clean-environment smoke: executed by a fresh venv's interpreter after installing the built wheel by path — never the repository. It asserts the import resolves under site-packages rather than a shadowing checkout, that `importlib.metadata` and `engine_version()` both report the expected version, that the class catalog loads (the data shipped), that the quickstart path runs end-to-end — characters, party, a minimal dungeon, a `GameSession`, commands, `format_message` over the events, and a save/load round-trip — and that the shipped OGL text is readable via `importlib.resources`. The script imports nothing from `tests/` or `tools/`.
- Not pytest, pinned: pytest runs against the repository tree; these run against artifacts and take a path and a version. Wiring them into the test suite would invert that.
- The workflow trigger, pinned: `release.yml` fires on pushed tags matching `v*`, and the tag format is `vX.Y.Z` — the GitHub convention, unambiguous in the trigger glob, and what `gh release` and compare links expect. Its first job fails fast if the tag does not equal `v` + the pyproject version, so a tag on an unbumped tree dies before any matrix spins up.
- Re-run the gate, don't gate on CI, pinned with its mechanism: `ci.yml`'s push trigger filters to `main`, so a tag push triggers no CI run — there is no run "for the tag" to wait on — and interrogating the merge commit's check runs would couple the release to CI bookkeeping and merge timing. The release workflow re-runs the full standing gate itself — the test matrix on both OSes, the strict docs build (no Pages upload), and the srd-data drift check — duplicated from `ci.yml` with a cross-reference comment, pinned over a `workflow_call` refactor: some thirty duplicated lines against refactoring a working Pages pipeline for an invisible benefit.
- The job graph after the gate, pinned: build once on ubuntu — `uv build`, `check_dist.py`, upload `dist/` as the artifact — then install-smoke on both OSes against the downloaded artifact (the wheel is pure Python, but the smoke is cheap and mirrors the support matrix CI promises: one build, two install proofs), then publish, then the GitHub Release. What publishes is byte-for-byte what was audited and smoked.
- Publishing, pinned: `pypa/gh-action-pypi-publish` under OIDC trusted publishing — a dedicated `pypi` GitHub environment, `id-token: write` on the publish job alone, and no credentials anywhere in the repository. The publish job is the last state-changing step before the release job, which extracts the tag's section from `CHANGELOG.md` and creates the GitHub Release with those notes verbatim.
- The manual prerequisite, the loudest callout in this plan: before the tag is pushed, the owner must, on pypi.org, add a pending trusted publisher — PyPI project name `osrlib`, owner `mmacy`, repository `osrlib-python`, workflow `release.yml`, environment `pypi` — and create the `pypi` environment in the repository settings (no required reviewers by default: this is a solo project and the tag push is already the human trigger). The pending publisher is also what claims the unregistered name atomically at first publish.
- The recovery rule, pinned: any failure before the publish job leaves PyPI untouched — delete the tag, fix on a branch, re-tag. Once publish succeeds, that version's filenames are burned on PyPI and the next attempt is a new version. This rule goes in the runbook.
- Considered and rejected, pinned: pinning the setup-uv action or the uv frontend version in the workflow — `[build-system]` already bounds `uv_build`, and the backend, not the frontend, is the reproducibility control.

### 5. The install-instruction swap — `README.md`, `docs/index.md`, `docs/getting-started/quickstart.md`

- The swap, pinned: the README status line drops "pre-release … no PyPI release yet" for a released statement, and the installation sections in all three files move to PyPI instructions carrying both spellings — `uv add osrlib` (the repository's own toolchain) and `pip install osrlib` (the universal one). The from-source path moves to its actual audience: the README's contributing section gains the clone-and-`uv sync` lines the installation section drops — after the swap, contributors are the only readers who install from source, and theirs becomes the only section that says how.
- The twin gate stands guard, pinned: the README's quickstart block is byte-identical to the package docstring's by the Phase 7 equality gate, so this work item does not touch the quickstart block — and the gate staying green is the explicit check that it didn't.
- The ordering subtlety, pinned as accepted: merging the implementation PR redeploys the docs site claiming PyPI availability minutes before the tag-driven publish completes. The alternatives — a two-stage docs PR, or holding the docs deploy — buy a second review loop to close a window whose audience is approximately nobody. The release checklist closes it instead: merge, then tag the merge commit immediately.

### 6. The versioned-docs decision and the releasing runbook — `AGENTS.md`

- Versioned documentation, decided as Phase 7 deferred: not at 1.0, pinned. Pages-from-`main` remains the whole deployment. A version switcher with exactly one released version is chrome, and adopting mike means carrying its deploy model for zero reader benefit. The adoption trigger, named so it cannot silently slip: the first post-1.0 release whose published docs must describe behavior different from `main` — practically, the first minor or major with behavior or API changes — adopts mike or equivalent in that release's own plan; patch and docs-only releases do not trigger it.
- Implementation amendment: the plan was silent on one sentence this phase falsifies — the greenfield-discipline section's "osrlib is pre-1.0 with no external consumers" premise in `AGENTS.md`. The section recalibrates: the frozen public surface under semantic versioning joins `schema_version` and determinism as the real compatibility contracts, and the internal refactor-outright discipline stands unchanged behind that surface.
- The runbook, pinned: `AGENTS.md` gains a short "Releasing" section — the version lives in `pyproject.toml` alone and `engine_version()` reads installed metadata; the bump procedure is edit, `uv lock`, and nothing else, because the golden comparisons normalize the stamp and regenerating a golden for a version bump is a defect; user-visible changes add their `[Unreleased]` changelog bullet in the same PR; releases are an annotated `vX.Y.Z` tag on the merge commit, and the workflow gates, builds, audits, publishes, and cuts the GitHub Release with no tokens in play; the local dry run is `uv build`, then `check_dist.py`, then `install_smoke.py` in a fresh venv; the recovery rule above; and the versioned-docs adoption trigger from the previous bullet, recorded in the runbook because post-1.0 releases are not roadmap phases and nothing else would send a future release-runner back to this plan. Releasing is now a repeating procedure with no other home, and the golden-regeneration trap in particular must be written down.

## Sequencing

1. Work items 1 and 3 together: the tree claims 1.0.0 with complete metadata, the lockfile follows, and the suite is green under the normalized comparisons — everything downstream builds and audits this artifact.
2. Work item 2: the changelog exists before the workflow that extracts a section from it.
3. Work item 4, scripts before workflow: the audits are proven by the local dry run — `uv build`, `check_dist.py` against the fresh `dist/`, `install_smoke.py` in a fresh venv — and only then wired into `release.yml`, which cannot itself be exercised until a tag exists.
4. Work item 5 last among content changes, so no intermediate state of the branch documents an install path falser than the one it replaces.
5. Work item 6 and the closing pass: the runbook, the release checklist finalized, and every deferral in this plan verified against its named pickup.
6. After merge: the release checklist below — it, not the PR, finishes the phase.

## Definition of done

- `uv sync && uv run ruff format --check && uv run ruff check && uv run pyright && uv run pytest && uv run mkdocs build --strict` passes locally and in CI on both OSes plus the docs and srd-data jobs, with `uv sync --locked` satisfied by the refreshed lockfile.
- The golden files are byte-identical and unregenerated, `SCHEMA_VERSION` is still 2, the phase's `src/` diff is empty, and the only standing-test diffs are the two stamp normalizations — both still asserting the live stamp equals `engine_version()`.
- `pyproject.toml` carries version 1.0.0, the license expression and both license files, the pinned classifiers, URLs, and keywords; `CHANGELOG.md` exists with the capability-bulleted 1.0.0 entry and the `[Unreleased]` scaffold.
- The local dry run is recorded in the implementation PR: `uv build` produced the sdist and wheel, `check_dist.py` passed against them, and `install_smoke.py` passed in a fresh venv against the wheel.
- `release.yml` and both `tools/release/` scripts are in the tree; the README, docs index, and quickstart carry the PyPI instructions with the twin-quickstart gate green; the FastAPI example README's sample handshake says 1.0.0; `AGENTS.md` carries the releasing runbook.
- Every pinned decision in this plan is recorded here and reflected in the shipped configuration, `docs/adaptations.md` gains no entries, and every deferred item names its pickup.

## Release checklist

The definition of done above is what the implementation PR can prove before a tag exists; publication itself is this checklist, executed by the owner, in order. The phase is complete when it has run to the end.

1. On pypi.org (an account with 2FA): add the pending trusted publisher — project `osrlib`, owner `mmacy`, repository `osrlib-python`, workflow `release.yml`, environment `pypi`. In the repository settings, create the `pypi` environment.
2. Merge the implementation PR.
3. Immediately after the merge: `git tag -a v1.0.0 -m "osrlib 1.0.0"` on the merge commit, then `git push origin v1.0.0`.
4. Watch `release.yml` through the publish and release jobs. On any failure before publish: delete the tag, fix on a branch, re-tag. After a successful publish, a failed release job is recoverable by hand (`gh release create`); the publication stands.
5. Verify: the PyPI project page renders the README, license, and URLs; `pip install osrlib` in a clean venv imports and reports `engine_version() == "1.0.0"`; the GitHub Release exists with the changelog notes; and the live docs site's install instructions are now true.
