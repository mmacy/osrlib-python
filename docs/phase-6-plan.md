# Phase 6 plan — API freeze

Implementation plan for Phase 6 of [the osrlib spec](spec.md). Phase 6 freezes the public API: the polish audit that spends the last free-rename window before anything downstream depends on the surface, the Advanced Fantasy groundwork validation the data model has promised since Phase 0 (exercised against hand-authored fixture classes and a fixture spell, and opening the two rigidities the audit found — the closed `Race` enum and the closed spell-list vocabulary), and the FastAPI wrapper example the spec assigns to this phase (the per-session lock demonstration, player-visibility filtering at the wire, and the seed-is-a-server-secret discipline) standing beside the Phase 5 TUI as the second front-end proof. The roadmap's two remaining phases build on the frozen surface: Phase 7 documents it (the docstring overhaul and the documentation site) and Phase 8 releases it (packaging, PyPI). The milestone: after this phase, the public surface is final — every rename done, the groundwork validated, both example front ends running against the same engine, and a type checker's word for the `py.typed` claim.

Phase 6 is a no-new-rules phase. Nothing here touches an RNG path, a draw order, an event code, a serialized field, or the package version: the golden files stand byte-for-byte, `SCHEMA_VERSION` stays 2, and `docs/adaptations.md` gains no rules entries. Every pin in this plan is an engineering decision, recorded here rather than in the register.

## Scope

In scope:

- Advanced Fantasy groundwork: the data-model audit, `race` and `SpellTemplate.spell_list` opened from closed vocabularies to validated string ids (the audit's two genuine rigidities — no mechanic consumes race, and the spell side's one procedural consumer, the divine reversal-at-cast key, re-keys onto the caster profile with the opening), fixture Advanced-shaped classes and a fixture spell run through the full kernel lifecycle behind the one declared test seam
- API polish: the public-surface census (`__all__` audit, naming and signature sweep with renames executed now, stale phase-promise language retired), the pinned import contract (one home per symbol, no re-exports anywhere), and a pyright basic-mode CI gate over `src/` and `examples/`
- `examples/fastapi_crawler/` — the FastAPI wrapper over `GameSession` and the Phase 5 barrow content: per-session lock, sync endpoints, player-visibility filtering, in-fiction rejections as 200s, typed errors mapped to HTTP statuses, save-by-id with the save document never leaving the server
- Tests: the groundwork lifecycle suite, wire-leak assertions for the FastAPI example, the `__all__`-completeness and pyright gates — with the standing suites untouched and green

Out of scope (deferred to the phase or track that picks each up):

- The documentation pass — the docstring overhaul to shippable new-user quality (chartered by the audit in [docs/docstring-audit.md](docstring-audit.md)), the mkdocs-material + mkdocstrings site with its strict-build cross-reference checking, the guides and both front-end walk-throughs (including work item 1's custom-classes authoring guide and work item 3's FastAPI pattern page), the generated command/event JSON Schema reference, and the README rewrite — is Phase 7, deliberately after this phase so the docs describe a surface that no longer moves. Phase 6 owes docstrings accuracy (the stale-promise sweep); Phase 7 owes them quality.
- Release engineering and publication — version 1.0.0 and its one test consequence (the engine-version stamp normalization in the phase 4 and 5 golden comparisons), `CHANGELOG.md`, pyproject metadata completion, the wheel-content audit and clean-environment install smoke, the tag-driven trusted-publishing workflow, and the PyPI publication itself (the `osrlib` name is unregistered as of this writing — the JSON API 404s) — is Phase 8. The package version does not move in Phase 6.
- Advanced Fantasy content — compiling Advanced classes, races, spells, and monsters is post-1.0 work that lands through the SRD pipeline into the shipped catalogs; this phase validates the landing zone, nothing more
- A runtime content-injection API (games registering custom classes with a live session) — pinned as a deliberate boundary, not a gap: the spec's extension points are listeners and flags, catalog content arrives as compiled data, and the groundwork fixtures exercise the definition-parameterized kernel without inventing a registration surface; post-1.0 if a consumer ever demonstrates the need
- Wilderness and sea adventuring, strongholds and domain play, retainers, magical research, procedural dungeon generation — the spec's own post-1.0 list, unchanged
- An async facade — never in-package per the spec's technology decisions; the FastAPI example demonstrates the sync-wrapper pattern instead
- Production backend concerns in the FastAPI example (authentication, multi-process session stores, persistence beyond the in-memory save store) — the example teaches the lock and visibility patterns, not deployment
- The `model_construct` hot-path fast lane — the spec sanctions it only when profiling shows event construction dominating mass simulations; no consumer has demonstrated that, so it stays unexercised
- An LLM-referee example front end — the spec names it a first-class consumer and Phase 7's docs ship the command/event schemas it would consume, but an example agent is post-1.0

## Work items

### 1. Advanced Fantasy groundwork — `core/classes.py`, `core/character.py`, `core/spells.py`, `tests/test_advanced_groundwork.py`

- The audit, recorded: the spec's flavor strategy promises Advanced classes as additive data — class definitions are pure data, `character_class` and `race` are separate fields, race is implied by class in Classic play. The model holds everywhere but two places. First, `Race` is a closed `StrEnum` (human, dwarf, elf, halfling), so a gnome or half-orc is a code edit — yet nothing mechanical reads race: infravision, detection checks, ghoul-paralysis immunity, and the halfling bonuses all resolve through structured class-ability tags, and the only consumers of `.race` copy it from the class definition into the character. Race is identity data, and a closed enum is the wrong shape for it. Second, `SpellTemplate.spell_list` is `Literal["cleric", "magic_user"]` while its class-side counterpart (`CasterProfile.spell_list`, read from the caster tag's params) is already an open string — legality is a string match, and the closed side of it forecloses exactly the Advanced pressure the flavor strategy names, since Advanced divine and arcane casters bring their own lists. What stays closed is pinned with the line that separates it: armour/weapon policy kinds and other closed procedure vocabularies are kernel semantics — the spec's "rules procedures are code" — so a new policy kind is legitimately a code addition, while race is an identity string with no procedural meaning of its own and spell-list has exactly one procedural consumer, named and re-keyed below: `_memorized_index`'s divine choose-reversal-at-cast freedom branches on the literal `"cleric"` list name as a proxy for the caster's kind (its memorization-side twin already keys on `profile.kind`, and the generation-side literal in `core/treasure.py` merely picks between the two Classic lists and stays). The spell-level bound (`le=6`) matches the Advanced roster's own ceiling, so widening it is deferred until real content demands it.
- `race` opens to a validated string id, pinned: `race: str` with pattern `^[a-z][a-z0-9_]*$` on both `ClassDefinition` and `Character`; the `Race` enum is deleted outright (greenfield discipline — no dead enum kept for old paths); the compiled Classic classes carry exactly the four existing wire values, asserted in the data tests. `spell_list` opens the same way on `SpellTemplate`, pattern-validated, with the two existing wire values asserted in the spell data tests — and its one procedural consumer re-keys with it, pinned: `_memorized_index`, and through it `validate_cast` and `cast_spell`, takes the caster's `CasterProfile`, supplied by call sites from the definition they already hold (the `memorize_spells` precedent; callers that skip the memorized check pass none), and the divine freedom keys on `profile.kind == "divine"` instead of the list name. Divine reversal-at-cast then works on every divine list — the exact behavior the Advanced landing zone exists to enable, and one the cleric-literal key would silently deny every non-cleric divine caster. No `SCHEMA_VERSION` bump for either opening, pinned with the reading stated: every existing version-2 document remains valid (the enum and Literal serialized as their string values, which the open fields accept unchanged), nothing is renamed, removed, or re-meant, and widening a field's accepted domain is the write-side mirror of the additive-schema rule that consumers must tolerate unknown values — the same reading that lets new event types ship without a bump. Pre-1.0 there are no external artifacts to break; 1.0's schema contract begins with the fields already open.
- The fixtures, pinned as hand-authored test data — invented mechanics in B/X shape, not scraped, not shipped, so the OGL question never arises: a human "warden" (divine half-caster with spell slots from level 1 — a progression shape no Classic class has — its own save table, requirements, two-tier prime requisites, and a caster tag whose params name spell list `warden`) and a "gnome" (race id `gnome` exercising the opened field, max level 8, demi-human saves, and class-ability tags drawn from the compiled Classic census — infravision with its range param and a detection check — so the fixture exercises the same tag consumers the shipped classes do). A reversible fixture spell carries `spell_list="warden"`, proving the spell side is additive the same way and giving the reversal re-keying its test subject. Fixture definitions are built as models and additionally round-tripped through JSON into `ClassCatalog`/`SpellCatalog` validation — the identical validation path the shipped loaders apply.
- The resolution boundary, pinned: the stepwise kernel functions take a `ClassDefinition` directly, but `Character` itself resolves the shipped catalog — its structural validator calls `Character.definition`, which reads the module-level `load_classes` binding in `core/character.py`, on every construction, every `model_validate`, and (under `validate_assignment`) every field assignment. A fixture-class character therefore cannot exist without a seam. The seam, pinned as one binding: work item 2's session dedup routes the session's four inline `load_classes().get(member.class_id)` lookups through `Character.definition`, leaving `core/character.py`'s `load_classes` name the lifecycle path's single class-catalog resolution point (the kernel's two other resolvers — the NPC generator's and `minimum_caster_level`'s — sit on no lifecycle step and stay as they are), and the fixture suite monkeypatches exactly that binding with a loader returning the shipped catalog extended by the fixtures. One declared patch covers construction, mutation-triggered revalidation (`level_up`, memorization), and `party_from_document` alike. This is a test seam, not a shipped surface — the out-of-scope no-injection-API pin stands, and the groundwork claim is honest about it: catalog content ships through the pipeline; the seam exists so test content can exercise the same validation the shipped catalog gets.
- The lifecycle validation: ability-score rolls and `validate_class_choice` against the fixture requirements, the adjustment step honoring `may_not_lower`, hit points, starting spells for the warden, equip legality under the fixture armour/weapon policies, `level_up` through the row recompute (slots appearing per the fixture progression), `drain_levels` as its inverse, saves and attack values off the fixture rows, memorization and casting legality for the fixture spell on the `warden` list — memorized unreversed and cast reversed, locking the divine choose-at-cast freedom on a non-cleric list — and the character document (class id `warden`, race `gnome`) stamping and round-tripping through `party_from_document` — all under the declared seam.
- The developer-facing authoring guide that walks this same path is Phase 7 documentation; in Phase 6 the fixture suite is the groundwork proof, and the suite's docstrings state the path plainly enough to seed that guide.

### 2. API polish — every public module

- The census: every public module carries a complete `__all__` (audited against its public defs), every public symbol a house-style docstring (ruff's pydocstyle gate already enforces presence; the sweep audits accuracy), and the naming/signature pass runs now — this is the last window where a rename is free, and greenfield discipline says take it. The freeze boundary, pinned: code symbols (functions, classes, parameters, module paths) rename freely this phase; serialized names — event codes, rejection codes, command `command_type` strings, model field names, enum wire values — are schema-governed, frozen at version 2, and no planned polish touches one. A polish finding that wants a serialized rename is out of scope by definition.
- The docstring boundary against Phase 7, pinned: Phase 6 owes every docstring accuracy — no stale claims, no dangling forward references, correct Args/Returns/Raises — because renamed and re-signed symbols must not ship lies. Phase 7 owes the quality overhaul: new-user readability, examples, orientation. The sweep here fixes what the rename pass touches and what the stale-promise hunt finds; it does not attempt Phase 7's rewrite.
- The import contract, pinned: one home per symbol, no convenience re-exports anywhere — which means the four existing top-level re-exports (`OsrlibError`, `ContentValidationError`, `SCHEMA_VERSION`, `engine_version` in `osrlib/__init__.py`) are cut, the `Alignment` precedent applied to the package root itself; `osrlib/__init__.py` becomes docstring-only like `core/` and `crawl/`, the Phase 7 docs will teach `osrlib.errors` and `osrlib.versioning` as those symbols' homes, and the standing `test_top_level_exports` guard retires with the re-exports it guards. A curated top-level namespace (`osrlib.GameSession`) was considered and rejected: it creates dual import paths for every exported symbol, which is the ambiguity AGENTS.md bans, and Phase 7's documentation site is the discovery surface a flat namespace would badly imitate.
- The resolution dedup: the session's four inline `load_classes().get(member.class_id)` lookups — three `GameSession` methods and the `_handle_award_xp` handler — become `member.definition`, and the module's `load_classes` import goes with them: duplicate resolution logic replaced by the accessor that already exists, on its own merits, and incidentally the change that gives work item 1 its single-binding seam.
- The stale-promise sweep: six phases of docstrings and comments carry forward-references ("arrives in Phase 4", "Phase 5's award will drive this") that are now history; they reword to timeless statements or delete. TODO/FIXME count in `src/` must end at zero — anything real becomes a named out-of-scope entry with its pickup track, anything else dies.
- The typing gate, pinned: pyright in basic mode over `src/` and `examples/` joins CI. The codebase has been fully annotated under house style since Phase 0 and ships `py.typed`; a library that claims `Typing :: Typed` should have a checker's word for it. Findings are fixed, not suppressed; a pydantic idiom the checker cannot see through may carry a targeted `# type: ignore[code]` with a reason. If implementation reveals structural noise beyond that, the plan amends with the decision rather than shipping a suppression blanket.
- The exception audit: every raise path in `src/` uses the typed `errors.py` hierarchy (the FastAPI example's status mapping consumes this), and rejection codes stay total — every rejection path emits a code `messages.py` can format.

### 3. The FastAPI example — `examples/fastapi_crawler/`

- A FastAPI app wrapping `GameSession`, serving the Phase 5 barrow: `content.py` imports from `examples.tui_crawler.content` unchanged — the same adventure behind a terminal and an HTTP API is the spec's presentation-agnostic claim made concrete. Dependencies land in a new `examples` dependency group (`fastapi`, `httpx` for the TestClient), included by the `dev` group so `uv sync` stays one command; the package's runtime dependencies remain pydantic alone, per the spec.
- Endpoints are sync `def`, pinned: FastAPI runs sync handlers in its threadpool, which is precisely why the per-session lock exists and is demonstrated — an in-memory session store holding `(GameSession, threading.Lock)` per session id, the lock held across every `execute` and view read. The spec names this exact demonstration as Phase 6's. A concurrency test posts commands from racing threads and asserts serialization (every command accepted or rejected cleanly, the event log a valid interleaving, no corruption) without asserting cross-thread order, which no lock promises.
- The surface, pinned: `POST /sessions` creates a session from a stamped party document and optional client seed — by default the server draws the seed and never returns it (the spec: full state, referee events, and the master seed are server-side secrets; the client-supplied seed exists for reproducible demos and the tests). `GET /sessions/{id}` returns session metadata — the schema handshake (`schema_version`, engine version) the spec promises front ends. `POST /sessions/{id}/commands` parses the body through `parse_command` (an unknown command type is a 422), executes under the lock, and returns the result with events filtered to player visibility — referee events are stripped at the wire, never trusted to the client. `GET /sessions/{id}/view` returns `session.view(Visibility.PLAYER)` and there is no referee-view endpoint at all. `POST /sessions/{id}/save` snapshots into a server-side save store and returns a save id — the save document (seed, referee state, full logs) never crosses the wire — and `POST /sessions` accepts a save id to restore.
- Status mapping, pinned to mirror the spec's error split: an in-fiction rejection (`accepted: false`) is a 200 — rejections are game feedback, not transport errors; out-of-fiction failures map from the typed hierarchy — `ContentValidationError` 422, `SaveVersionError` 409, unknown session or save id 404, anything else 500. This mapping is the `errors.py` design earning its keep, and the example documents it as the pattern.
- Tests: a TestClient scripted playthrough (a shortened barrow script — create, delve, fight, loot, return, award), the concurrency test, the handshake, the status mapping, and the wire-leak suite — the Phase 5 leak property test's assertions applied to serialized responses: no response body from any endpoint ever contains the seed, a referee-visibility event, unexplored geometry, monster HP, hidden charges, or an unidentified item's true identity. The example's README documents the run (`uv run uvicorn examples.fastapi_crawler:app`) and each pattern; the site walk-through built on it is Phase 7's.

### 4. Tests

- **Groundwork**: the work item 1 lifecycle suite — fixture catalogs validating through the loader models, the warden and gnome through creation, adjustment, equip policy, leveling, drain, saves, memorization and casting (the reversed cast on the `warden` list locking the re-keyed divine freedom), and the document round-trip on the opened race field, all under the single declared loader-binding seam; the data tests asserting the compiled Classic classes carry exactly the four legacy race strings and the compiled spells exactly the two legacy list strings; a regression case asserting cleric reversal-at-cast behaves identically before and after the re-keying.
- **API polish**: the pyright CI gate; an `__all__`-completeness test (public defs vs. declared exports per module); the existing exhaustive-template test standing guard over message-code totality.
- **FastAPI example**: the scripted TestClient playthrough, the racing-threads serialization test, the schema handshake, the status-mapping table case-by-case, and the wire-leak suite over every endpoint's serialized responses.
- **Standing suites**: every existing test passes, with exactly one sanctioned test change — `test_top_level_exports` retires with the re-exports it guards (work item 2). (Mechanical call-site updates from work item 2's signature sweep don't count against this census — those tests keep their assertions; the sanctioned change is the one whose asserted surface itself dies.) The golden files are byte-identical and unregenerated (no RNG path and no version stamp moves this phase), the schema stays at version 2 with no new migration, and the leak property tests extend only by the wire-leak reuse. Any golden diff, and any schema diff, in this phase's changes is a defect by definition.

## Sequencing

1. Advanced groundwork first: the race and spell-list openings, the reversal re-keying, the session resolution dedup (work item 2's change, pulled forward because it is the fixture seam), and the fixture suite land before anything freezes — this is the one item allowed to move model code, so it goes while the window is open.
2. API polish: the census, renames, stale-promise sweep, the re-export cut, and the pyright gate — the surface settles here.
3. The FastAPI example against the settled surface (its consumer pressure is the last polish check — friction it hits reopens item 2 findings before the freeze is declared).
4. The closing pass: every deferral in this plan verified against its pickup phase (7 or 8), the README status line updated, and the freeze declared — Phase 7 documents what this phase froze.

## Definition of done

- `uv sync && uv run ruff format --check && uv run ruff check && uv run pytest` passes locally and in CI on both OSes, with the new gates — pyright basic and the `__all__`-completeness test — green beside the standing suites, the golden files byte-identical and unregenerated, `SCHEMA_VERSION` still 2, and the package version untouched at 0.1.0.
- The groundwork holds: `race` is an open validated string on `ClassDefinition` and `Character` with the `Race` enum gone, `spell_list` an open validated string on `SpellTemplate` with the `Literal` gone, and the four Classic race values and two Classic list values asserted in data tests; the warden and gnome fixtures and the reversible `warden`-list fixture spell validate through the loader models and complete the full lifecycle — creation, adjustment, equip policy, leveling, drain, memorization, casting including the reversed cast on a non-cleric divine list, document round-trip — under the single declared `core/character.py` loader-binding seam, with no library change beyond the two openings, the reversal re-keying onto `CasterProfile.kind`, and the session resolution dedup.
- The public surface is frozen: every public module has a complete `__all__` and accurate docstrings, the four top-level re-exports are cut and `osrlib/__init__.py` is docstring-only, no forward-referencing phase language survives in `src/`, TODO/FIXME count is zero, and no serialized name changed.
- The FastAPI example serves the barrow with sync handlers under per-session locks; rejections return 200 with the rejection payload while typed errors map 422/409/404/500; the seed and save documents never cross the wire; the player view and player-visibility events are the only game state any endpoint returns; the scripted playthrough, concurrency, handshake, and wire-leak tests are green; the TUI example runs unchanged.
- Every pinned decision in this plan is recorded here and reflected in the shipped configuration; `docs/adaptations.md` gains no entries, because Phase 6 pins engineering, not rules; and every deferred item names Phase 7, Phase 8, or its post-1.0 track.
