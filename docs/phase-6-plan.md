# Phase 6 plan — release

Implementation plan for Phase 6 of [the osrlib spec](spec.md), the final roadmap phase. Phase 6 ships osrlib 1.0.0: the API-polish audit that spends the last free-rename window before the public surface freezes, the Advanced Fantasy groundwork validation the data model has promised since Phase 0 (exercised against hand-authored fixture classes and a fixture spell, and opening the two rigidities the audit found — the closed `Race` enum and the closed spell-list vocabulary), the FastAPI wrapper example the spec assigns to this phase (the per-session lock demonstration, player-visibility filtering at the wire, and the seed-is-a-server-secret discipline), the documentation site (mkdocs-material with mkdocstrings, turning six phases of house-style docstring cross-references into checked links), and the release engineering that puts `osrlib` on PyPI under trusted publishing with the OGL data licensing intact in the wheel. The spec's Phase 6 entry names the deliverables directly — API polish, documentation pass with example front-end usage (sync TUI, FastAPI wrapper), publication to PyPI as `osrlib`, Advanced Fantasy groundwork validated against the data model — and the milestone is the release itself: `uv add osrlib` in a clean project yields a working, documented, typed rules engine.

Phase 6 is a no-new-rules phase. Nothing here touches an RNG path, a draw order, an event code, or a serialized field: the golden files stand byte-for-byte unregenerated (the 1.0.0 engine-version stamp is the release's one run-to-run difference, normalized where golden comparisons check it — work item 7), `SCHEMA_VERSION` stays 2, and `docs/adaptations.md` gains no rules entries. Every pin in this plan is an engineering decision, recorded here rather than in the register.

## Scope

In scope:

- Advanced Fantasy groundwork: the data-model audit, `race` and `SpellTemplate.spell_list` opened from closed vocabularies to validated string ids (the audit's two genuine rigidities — no mechanic consumes race, and the spell side's one procedural consumer, the divine reversal-at-cast key, re-keys onto the caster profile with the opening), fixture Advanced-shaped classes and a fixture spell run through the full kernel lifecycle behind the one declared test seam, and the custom-class authoring guide that doubles as the groundwork proof
- API polish: the public-surface census (`__all__` audit, naming and signature sweep with renames executed now, stale phase-promise language retired), the pinned import contract (one home per symbol, no top-level re-exports), and a pyright basic-mode CI gate over `src/` and `examples/`
- `examples/fastapi_crawler/` — the FastAPI wrapper over `GameSession` and the Phase 5 barrow content: per-session lock, sync endpoints, player-visibility filtering, in-fiction rejections as 200s, typed errors mapped to HTTP statuses, save-by-id with the save document never leaving the server
- The documentation site: mkdocs-material + mkdocstrings under a `docs` dependency group, strict builds in CI, GitHub Pages deployment, the guide and reference set pinned in work item 4, docs-census completeness tests, and the PyPI-facing README rewrite
- Release engineering: version 1.0.0, `CHANGELOG.md`, completed pyproject metadata, the wheel-content audit and clean-environment install smoke test in CI, and the tag-driven release workflow publishing to TestPyPI then PyPI via trusted publishing
- Tests: the groundwork lifecycle suite, wire-leak assertions for the FastAPI example, docs completeness, packaging audits — with the standing suites untouched and green

Out of scope (deferred to the track that picks each up):

- Advanced Fantasy content — compiling Advanced classes, races, spells, and monsters is post-1.0 work that lands through the SRD pipeline into the shipped catalogs; this phase validates the landing zone, nothing more
- A runtime content-injection API (games registering custom classes with a live session) — pinned as a deliberate boundary, not a gap: the spec's extension points are listeners and flags, catalog content arrives as compiled data, and the groundwork fixtures exercise the definition-parameterized kernel without inventing a registration surface; post-1.0 if a consumer ever demonstrates the need
- Wilderness and sea adventuring, strongholds and domain play, retainers, magical research, procedural dungeon generation — the spec's own post-1.0 list, unchanged
- An async facade — never in-package per the spec's technology decisions; the FastAPI example demonstrates the sync-wrapper pattern instead
- Production backend concerns in the FastAPI example (authentication, multi-process session stores, persistence beyond the in-memory save store) — the example teaches the lock and visibility patterns, not deployment
- The `model_construct` hot-path fast lane — the spec sanctions it only when profiling shows event construction dominating mass simulations; no consumer has demonstrated that, so it stays unexercised
- An LLM-referee example front end — the spec names it a first-class consumer and the docs ship the command/event schemas it would consume, but an example agent is post-1.0
- Docs versioning (mike) and localization of `messages.py` — single-version English docs at 1.0; message localization is front-end territory per the spec

## Work items

### 1. Advanced Fantasy groundwork — `core/classes.py`, `core/character.py`, `core/spells.py`, `tests/test_advanced_groundwork.py`

- The audit, recorded: the spec's flavor strategy promises Advanced classes as additive data — class definitions are pure data, `character_class` and `race` are separate fields, race is implied by class in Classic play. The model holds everywhere but two places. First, `Race` is a closed `StrEnum` (human, dwarf, elf, halfling), so a gnome or half-orc is a code edit — yet nothing mechanical reads race: infravision, detection checks, ghoul-paralysis immunity, and the halfling bonuses all resolve through structured class-ability tags, and the only consumers of `.race` copy it from the class definition into the character. Race is identity data, and a closed enum is the wrong shape for it. Second, `SpellTemplate.spell_list` is `Literal["cleric", "magic_user"]` while its class-side counterpart (`CasterProfile.spell_list`, read from the caster tag's params) is already an open string — legality is a string match, and the closed side of it forecloses exactly the Advanced pressure the flavor strategy names, since Advanced divine and arcane casters bring their own lists. What stays closed is pinned with the line that separates it: armour/weapon policy kinds and other closed procedure vocabularies are kernel semantics — the spec's "rules procedures are code" — so a new policy kind is legitimately a code addition, while race is an identity string with no procedural meaning of its own and spell-list has exactly one procedural consumer, named and re-keyed below: `_memorized_index`'s divine choose-reversal-at-cast freedom branches on the literal `"cleric"` list name as a proxy for the caster's kind (its memorization-side twin already keys on `profile.kind`, and the generation-side literal in `core/treasure.py` merely picks between the two Classic lists and stays). The spell-level bound (`le=6`) matches the Advanced roster's own ceiling, so widening it is deferred until real content demands it.
- `race` opens to a validated string id, pinned: `race: str` with pattern `^[a-z][a-z0-9_]*$` on both `ClassDefinition` and `Character`; the `Race` enum is deleted outright (greenfield discipline — no dead enum kept for old paths); the compiled Classic classes carry exactly the four existing wire values, asserted in the data tests. `spell_list` opens the same way on `SpellTemplate`, pattern-validated, with the two existing wire values asserted in the spell data tests — and its one procedural consumer re-keys with it, pinned: `_memorized_index`, and through it `validate_cast` and `cast_spell`, takes the caster's `CasterProfile`, supplied by call sites from the definition they already hold (the `memorize_spells` precedent; callers that skip the memorized check pass none), and the divine freedom keys on `profile.kind == "divine"` instead of the list name. Divine reversal-at-cast then works on every divine list — the exact behavior the Advanced landing zone exists to enable, and one the cleric-literal key would silently deny every non-cleric divine caster. No `SCHEMA_VERSION` bump for either, pinned with the reading stated: every existing version-2 document remains valid (the enum and Literal serialized as their string values, which the open fields accept unchanged), nothing is renamed, removed, or re-meant, and widening a field's accepted domain is the write-side mirror of the additive-schema rule that consumers must tolerate unknown values — the same reading that lets new event types ship without a bump. Pre-1.0 there are no external artifacts to break; 1.0's schema contract begins with the fields already open.
- The fixtures, pinned as hand-authored test data — invented mechanics in B/X shape, not scraped, not shipped, so the OGL question never arises: a human "warden" (divine half-caster with spell slots from level 1 — a progression shape no Classic class has — its own save table, requirements, two-tier prime requisites, and a caster tag whose params name spell list `warden`) and a "gnome" (race id `gnome` exercising the opened field, max level 8, demi-human saves, and class-ability tags drawn from the compiled Classic census — infravision with its range param and a detection check — so the fixture exercises the same tag consumers the shipped classes do). A reversible fixture spell carries `spell_list="warden"`, proving the spell side is additive the same way and giving the reversal re-keying its test subject. Fixture definitions are built as models and additionally round-tripped through JSON into `ClassCatalog`/`SpellCatalog` validation — the identical validation path the shipped loaders apply.
- The resolution boundary, corrected from the draft and pinned: the stepwise kernel functions take a `ClassDefinition` directly, but `Character` itself resolves the shipped catalog — its structural validator calls `Character.definition`, which reads the module-level `load_classes` binding in `core/character.py`, on every construction, every `model_validate`, and (under `validate_assignment`) every field assignment. A fixture-class character therefore cannot exist without a seam. The seam, pinned as one binding: work item 2's session dedup routes the session's four inline `load_classes().get(member.class_id)` lookups through `Character.definition`, leaving `core/character.py`'s `load_classes` name the lifecycle path's single class-catalog resolution point (the kernel's two other resolvers — the NPC generator's and `minimum_caster_level`'s — sit on no lifecycle step and stay as they are), and the fixture suite monkeypatches exactly that binding with a loader returning the shipped catalog extended by the fixtures. One declared patch covers construction, mutation-triggered revalidation (`level_up`, memorization), and `party_from_document` alike. This is a test seam, not a shipped surface — the out-of-scope no-injection-API pin stands, and the groundwork claim is honest about it: catalog content ships through the pipeline; the seam exists so test content can exercise the same validation the shipped catalog gets.
- The lifecycle validation: ability-score rolls and `validate_class_choice` against the fixture requirements, the adjustment step honoring `may_not_lower`, hit points, starting spells for the warden, equip legality under the fixture armour/weapon policies, `level_up` through the row recompute (slots appearing per the fixture progression), `drain_levels` as its inverse, saves and attack values off the fixture rows, memorization and casting legality for the fixture spell on the `warden` list — memorized unreversed and cast reversed, locking the divine choose-at-cast freedom on a non-cleric list — and the character document (class id `warden`, race `gnome`) stamping and round-tripping through `party_from_document` — all under the declared seam.
- The authoring guide (work item 4's custom-classes page) documents the same path the fixtures walk — that page is the developer-facing form of the groundwork proof, the Phase 5 fetch-quest precedent applied to content instead of game logic.

### 2. API polish — every public module

- The census: every public module carries a complete `__all__` (audited against its public defs), every public symbol a house-style docstring (ruff's pydocstyle gate already enforces presence; the sweep audits accuracy), and the naming/signature pass runs now — this is the last window where a rename is free, and greenfield discipline says take it. The freeze boundary, pinned: code symbols (functions, classes, parameters, module paths) rename freely this phase; serialized names — event codes, rejection codes, command `command_type` strings, model field names, enum wire values — are schema-governed, frozen at version 2, and no planned polish touches one. A polish finding that wants a serialized rename is out of scope by definition.
- The import contract, pinned: one home per symbol, no convenience re-exports anywhere — which means the four existing top-level re-exports (`OsrlibError`, `ContentValidationError`, `SCHEMA_VERSION`, `engine_version` in `osrlib/__init__.py`) are cut, the `Alignment` precedent applied to the package root itself; `osrlib/__init__.py` becomes docstring-only like `core/` and `crawl/`, the docs teach `osrlib.errors` and `osrlib.versioning` as those symbols' homes, and the standing `test_top_level_exports` guard retires with the re-exports it guards. A curated top-level namespace (`osrlib.GameSession`) was considered and rejected: it creates dual import paths for every exported symbol, which is the ambiguity AGENTS.md bans, and the documentation site (work item 4) is the discovery surface a flat namespace would badly imitate.
- The resolution dedup: the session's four inline `load_classes().get(member.class_id)` lookups — three `GameSession` methods and the `_handle_award_xp` handler — become `member.definition`, and the module's `load_classes` import goes with them: duplicate resolution logic replaced by the accessor that already exists, on its own merits, and incidentally the change that gives work item 1 its single-binding seam.
- The stale-promise sweep: six phases of docstrings and comments carry forward-references ("arrives in Phase 4", "Phase 5's award will drive this") that are now history; they reword to timeless statements or delete. TODO/FIXME count in `src/` must end at zero — anything real becomes a named out-of-scope entry with its pickup track, anything else dies.
- The typing gate, pinned: pyright in basic mode over `src/` and `examples/` joins CI. The codebase has been fully annotated under house style since Phase 0 and ships `py.typed`; a 1.0 that claims `Typing :: Typed` should have a checker's word for it. Findings are fixed, not suppressed; a pydantic idiom the checker cannot see through may carry a targeted `# type: ignore[code]` with a reason. If implementation reveals structural noise beyond that, the plan amends with the decision rather than shipping a suppression blanket.
- The exception audit: every raise path in `src/` uses the typed `errors.py` hierarchy (the FastAPI example's status mapping consumes this), and rejection codes stay total — every rejection path emits a code `messages.py` can format.

### 3. The FastAPI example — `examples/fastapi_crawler/`

- A FastAPI app wrapping `GameSession`, serving the Phase 5 barrow: `content.py` imports from `examples.tui_crawler.content` unchanged — the same adventure behind a terminal and an HTTP API is the spec's presentation-agnostic claim made concrete. Dependencies land in a new `examples` dependency group (`fastapi`, `httpx` for the TestClient), included by the `dev` group so `uv sync` stays one command; the package's runtime dependencies remain pydantic alone, per the spec.
- Endpoints are sync `def`, pinned: FastAPI runs sync handlers in its threadpool, which is precisely why the per-session lock exists and is demonstrated — an in-memory session store holding `(GameSession, threading.Lock)` per session id, the lock held across every `execute` and view read. The spec names this exact demonstration as Phase 6's. A concurrency test posts commands from racing threads and asserts serialization (every command accepted or rejected cleanly, the event log a valid interleaving, no corruption) without asserting cross-thread order, which no lock promises.
- The surface, pinned: `POST /sessions` creates a session from a stamped party document and optional client seed — by default the server draws the seed and never returns it (the spec: full state, referee events, and the master seed are server-side secrets; the client-supplied seed exists for reproducible demos and the tests). `GET /sessions/{id}` returns session metadata — the schema handshake (`schema_version`, engine version) the spec promises front ends. `POST /sessions/{id}/commands` parses the body through `parse_command` (an unknown command type is a 422), executes under the lock, and returns the result with events filtered to player visibility — referee events are stripped at the wire, never trusted to the client. `GET /sessions/{id}/view` returns `session.view(Visibility.PLAYER)` and there is no referee-view endpoint at all. `POST /sessions/{id}/save` snapshots into a server-side save store and returns a save id — the save document (seed, referee state, full logs) never crosses the wire — and `POST /sessions` accepts a save id to restore.
- Status mapping, pinned to mirror the spec's error split: an in-fiction rejection (`accepted: false`) is a 200 — rejections are game feedback, not transport errors; out-of-fiction failures map from the typed hierarchy — `ContentValidationError` 422, `SaveVersionError` 409, unknown session or save id 404, anything else 500. This mapping is the `errors.py` design earning its keep, and the example documents it as the pattern.
- Tests: a TestClient scripted playthrough (a shortened barrow script — create, delve, fight, loot, return, award), the concurrency test, the handshake, the status mapping, and the wire-leak suite — the Phase 5 leak property test's assertions applied to serialized responses: no response body from any endpoint ever contains the seed, a referee-visibility event, unexplored geometry, monster HP, hidden charges, or an unidentified item's true identity. The example's README documents the run (`uv run uvicorn examples.fastapi_crawler:app`) and each pattern.

### 4. The documentation site — `mkdocs.yml`, `docs/`, `.github/workflows/`

- mkdocs-material with mkdocstrings-python under a `docs` dependency group. The build runs `--strict`, pinned consequence included: every `[`symbol`][module.symbol]` cross-reference written since Phase 0 becomes a checked link, and the docs build fails on any that dangle — the docstring convention finally pays its toll both ways. API reference pages generate from the docstrings per module, following the package layout (`core/`, `crawl/`, the top-level modules).
- The page set, pinned. Getting started: install, a five-minute à la carte kernel tour (an attack roll, a treasure hoard), a `GameSession` quickstart. Concepts: the command/event model, determinism and replay, time and effects, visibility. Guides: running a session, persistence and migrations, extension points (the fetch quest as the worked example, per the spec's quest-tracker contract), authoring adventure content, and custom classes (work item 1's groundwork guide). Reference: the generated API pages, the command catalog, the event census with codes and visibility levels, the rejection-code catalog, the `Ruleset` flag table, and the adaptations register (the existing `adaptations.md`, included as-is). Front ends: a TUI walk-through of `examples/tui_crawler` and the FastAPI walk-through of work item 3. Licensing: the MIT/OGL split, the Section 15 notice, and the trademark statement.
- The JSON Schemas, pinned as generated-at-build: the spec promises that commands and events have stable JSON schemas an LLM can consume as tool definitions, so the reference delivers them — a docs-build hook (mkdocs gen-files) emits `model_json_schema()` for every command and event type into schema reference pages, generated at build time so they can never drift from the models, and the concepts page names `model_json_schema()` as the programmatic path a front end or agent harness calls directly.
- Completeness is tested, not promised: a docs-census test asserts every command type, every event message code, and every rejection code in the registries appears in the reference pages — the exhaustive-template test precedent applied to documentation. A missing entry fails pytest, not review.
- CI: a docs job builds strict on every PR; deployment to GitHub Pages (the official `actions/deploy-pages` flow) runs on push to main. The phase plans (`phase-0` through this one) and `spec.md` stay in the repo but off the site's nav — they are development history, not user documentation.
- The README rewrites for its PyPI-landing role: what osrlib is, install, a thirty-second example, links to the docs site, the licensing summary. The per-phase development-status paragraph retires in favor of the changelog.

### 5. Release engineering — `pyproject.toml`, `CHANGELOG.md`, `.github/workflows/release.yml`

- Version 1.0.0, pinned: the roadmap has scoped every phase against "1.0" since Phase 0, and Phase 6 is the release phase — 0.1.0 goes straight to 1.0.0, no 0.x public warm-up. `CHANGELOG.md` starts in Keep a Changelog format with a single 1.0.0 entry summarizing the engine by subsystem (what shipped, not the phase-by-phase history — that story lives in the plans).
- The version bump's one test consequence, pinned: `engine_version()` is stamped into every document, the phase 4 and phase 5 golden suites compare freshly stamped party documents byte-for-byte against goldens recorded under 0.1.0, and only the creation-goldens suite normalizes the stamp before comparing. The two lagging comparisons adopt the creation-goldens normalization — the stamp is version metadata, not a draw outcome, and the determinism contract itself scopes identical outcomes to identical engine versions. The golden files are not regenerated: every draw-dependent byte stands, and the normalization is a test-side acknowledgment that the engine version is the one field a release phase legitimately changes.
- pyproject metadata completes: classifiers (`Development Status :: 5 - Production/Stable`, `Intended Audience :: Developers`, `Topic :: Games/Entertainment :: Role-Playing`, `Typing :: Typed`, `Operating System :: OS Independent`, the Python 3.14 classifier), keywords, and `project.urls` (Homepage, Documentation → the Pages site, Repository, Changelog).
- The wheel audit, as a CI test: build with `uv build`, then assert the wheel contains every `data/*.json`, `data/LICENSE-OGL.md`, and `py.typed`, and contains nothing from `srd/`, `tests/`, `examples/`, or `docs/` — the OGL-data-ships-in-the-wheel claim the README already makes becomes a tested invariant. The sdist ships the package sources and licenses (the repo, not the sdist, is the regeneration surface — `srd/` and `tools/` stay out).
- The install smoke test, as a CI job: install the built wheel into a clean environment (no repo checkout on `sys.path`), import, load every catalog through the shipped loaders, run an à la carte roll and a minimal `GameSession` command. This is the test that catches a packaging miss the suite never can, because the suite runs against the source tree.
- The release workflow, pinned: triggered by tags matching `v*`; asserts the tag equals the pyproject version before anything else; runs the full gates (format, lint, pyright, pytest, SRD-drift, docs strict, wheel audit, install smoke); publishes via PyPI trusted publishing (OIDC, no long-lived tokens) — prerelease tags (`v1.0.0rc1`) publish to TestPyPI, final tags to PyPI behind a `pypi` GitHub environment with required review; and creates the GitHub Release with the changelog entry as its body. The `osrlib` name is unregistered on PyPI as of this writing (the JSON API 404s) — the rc-to-TestPyPI run exercises the whole path first, and the final publish claims the name.

### 6. The release — execution checklist

- Order, pinned: all gates green on main → `v1.0.0rc1` tag → TestPyPI publish verified by installing the rc from TestPyPI into a clean project and running the smoke sequence → `v1.0.0` tag → PyPI publish through the environment gate → verify `uv add osrlib` from a clean project → GitHub Release published → docs site live and linked. Any failure stops the line; a fix means a new rc, never a force-retag (tags are immutable release history).

### 7. Tests

- **Groundwork**: the work item 1 lifecycle suite — fixture catalogs validating through the loader models, the warden and gnome through creation, adjustment, equip policy, leveling, drain, saves, memorization and casting (the reversed cast on the `warden` list locking the re-keyed divine freedom), and the document round-trip on the opened race field, all under the single declared loader-binding seam; the data tests asserting the compiled Classic classes carry exactly the four legacy race strings and the compiled spells exactly the two legacy list strings; a regression case asserting cleric reversal-at-cast behaves identically before and after the re-keying.
- **API polish**: the pyright CI gate; an `__all__`-completeness test (public defs vs. declared exports per module); the existing exhaustive-template test standing guard over message-code totality.
- **FastAPI example**: the scripted TestClient playthrough, the racing-threads serialization test, the schema handshake, the status-mapping table case-by-case, and the wire-leak suite over every endpoint's serialized responses.
- **Docs**: the docs-census completeness test (commands, event codes, rejection codes present in the reference pages); the strict build itself as the cross-reference checker in CI.
- **Packaging**: the wheel-content audit and the clean-environment install smoke as CI jobs; the tag-equals-version assertion in the release workflow.
- **Standing suites**: every existing test passes, with exactly two sanctioned test changes — the phase 4 and phase 5 golden suites adopt the creation-goldens engine-version normalization (work item 5), because the 1.0.0 stamp is the release's one run-to-run difference, and `test_top_level_exports` retires with the re-exports it guards (work item 2). (Mechanical call-site updates from work item 2's signature sweep don't count against this census — those tests keep their assertions; the two sanctioned changes are the ones whose asserted surface itself moves or dies.) The golden files themselves are byte-identical and unregenerated (no RNG path moves this phase), the schema stays at version 2 with no new migration, and the leak property tests extend only by the wire-leak reuse. Any other golden diff, and any schema diff, in this phase's changes is a defect by definition.

## Sequencing

1. Advanced groundwork first: the race and spell-list openings, the session resolution dedup (work item 2's change, pulled forward because it is the fixture seam), and the fixture suite land before anything freezes — this is the one item allowed to move model code, so it goes while the window is open.
2. API polish: the census, renames, stale-promise sweep, and the pyright gate — the surface settles here.
3. The FastAPI example against the settled surface (its consumer pressure is the last polish check — friction it hits reopens item 2 findings before the docs describe them).
4. The documentation site: reference generation and guides against the final API, both front-end walk-throughs, census tests, CI docs job, README rewrite.
5. Release engineering: metadata, changelog, wheel audit, install smoke, the release workflow.
6. The release checklist end to end: rc, TestPyPI, 1.0.0, PyPI, verification, GitHub Release, docs live.
