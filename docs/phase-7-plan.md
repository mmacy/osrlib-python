# Phase 7 plan — documentation

Implementation plan for Phase 7 of [the osrlib spec](spec.md). Phase 7 documents the surface Phase 6 froze: the docstring overhaul to shippable new-user quality (chartered by the audit in [docs/docstring-audit.md](docstring-audit.md), whose counts this plan re-verified against post-Phase-6 `main`), the documentation site (mkdocs-material + mkdocstrings, strict builds, deployed to GitHub Pages), the generated references the spec promises front-end and LLM consumers (the command/event JSON Schemas, plus the rejection-code, message-code, stream-key, and content-id references the audit found missing), the guides and walk-throughs for both example front ends (including the named Phase 6 deferrals — the custom-classes authoring guide, the FastAPI pattern page, and the promise that the docs teach `osrlib.errors` and `osrlib.versioning` as those symbols' homes), and the README rewrite. Phase 8 releases what this phase documents. The milestone: after this phase, a developer who has never seen this repository can install osrlib, learn it from the published site, and build a working front end without reading library source — and the docs describe a surface that no longer moves.

Phase 7 is a docs-only phase, bound by the Phase 6 freeze. Nothing here renames a symbol, changes a signature, adds a public symbol, touches an RNG path or a serialized name, or moves the package version: the `src/` diff for the entire phase is docstring and comment hunks only, the golden files stand byte-for-byte, and `SCHEMA_VERSION` stays 2. Where documentation wants a code shape the code doesn't have — a rejection-code registry, a combatant protocol — the docs adapt to the code, never the reverse; both cases are pinned below. `docs/adaptations.md` gains no entries, because Phase 7 pins engineering, not rules.

## Scope

In scope:

- The documentation toolchain: a `docs` dependency group, the mkdocs-material + mkdocstrings site with strict builds, the generated API reference over the `__all__` surface, a docs CI job, GitHub Pages deployment from `main`, and the tested-examples harness
- The docstring overhaul: the audit executed — vocabulary purge, orientation module docstrings, runnable examples on the entry points, the cross-seam quickstart, the command contract (modes, rejection codes, emitted events) with mechanical gates, named combatant types in prose, id parameters linked to their loaders
- The generated references: command/event JSON Schema pages plus raw downloadable schemas, the rejection-code reference, the event message-code table, the RNG stream-key list, and the content-id indexes
- Guides and walk-throughs: getting-started pages (quickstart, building an adventure), the crawl/views/determinism/kernel/listeners/authoring/ruleset guides, the TUI and FastAPI walk-throughs, the LLM-referee page, the adaptations register on the site, and the licensing page
- The README rewrite
- Tests: the example harness, the twin-quickstart equality gate, the banned-vocabulary tripwire, the command-contract gates, the reference drift gates — with the standing suites untouched and green

Out of scope (deferred to the phase or track that picks each up):

- Release engineering — version 1.0.0, `CHANGELOG.md`, pyproject metadata completion (including the `Documentation` URL this phase's site creates), packaging audits, trusted publishing, and the install-instruction swap from source to PyPI — is Phase 8; until then every install instruction says "from source" honestly
- Versioned documentation (mike or equivalent) — a release-time concern; Phase 8 decides it when there is a second version to show
- The LLM-referee example front end — post-1.0 per the Phase 6 plan; this phase ships the schemas and the guide page such an agent would consume, not the agent
- Re-typing the duck-typed kernel parameters with a `Combatant` protocol or explicit unions — rejected, pinned: Phase 6 deliberately typed the combatant convention as `Any` and froze the surface; the spec's "named types with cross-references replacing duck-typed `object` prose" is satisfied at the prose level (work item 2), where the docstrings name and link the accepted types. A protocol would reopen the frozen surface to fix a documentation problem
- New public symbols in service of documentation (a runtime rejection-code registry, re-exports for doc convenience) — rejected, pinned: the surface is final and a hand-maintained registry drifting from ad-hoc call sites is the exact hazard; the docs side scans the source instead (work item 3)
- Localization infrastructure for the message catalog — the formatter's design (structured facts, message codes, templates outside the models) already supports it; documenting that design is in scope, shipping i18n tooling is not
- Custom site theming, search tuning, analytics, custom domains — material defaults ship; polish without a consumer is churn
- Docstrings in `tests/` and `tools/` — maintainer-facing, not part of the shipped surface; the `tests/**` pydocstyle ignore stands, and `tools/` is already D-clean and gets no overhaul

## Work items

### 1. The documentation toolchain — `pyproject.toml`, `mkdocs.yml`, `.github/workflows/ci.yml`, `docs/_scripts/`

- Dependencies, pinned: a new `docs` dependency group — `mkdocs`, `mkdocs-material`, `mkdocstrings[python]`, `mkdocs-gen-files`, `mkdocs-literate-nav` — included by the `dev` group so `uv sync` stays one command (the `examples` group precedent); `pytest-examples` joins `dev` directly (it is a test dependency, not a docs one). The package's runtime dependencies remain pydantic alone.
- `mkdocs.yml`, pinned: `site_name: osrlib`, `site_url: https://mmacy.github.io/osrlib-python/`, material theme at defaults (no custom theming this phase), `docs_dir: docs` with `exclude_docs` covering the design documents (`spec.md`, `phase-*-plan.md`, `docstring-audit.md`) — user-facing pages live in subdirectories (`getting-started/`, `guides/`, `front-ends/`, `reference/`) beside them; strict mode with the link/anchor validation options raised to their strictest settings, so the 153 mkdocstrings-style cross-reference occurrences (119 distinct targets) already in the docstrings become checked contracts the moment the site exists.
- The API reference, generated: a `mkdocs-gen-files` script (living in `tools/docs/` with the rest of the docs tooling — the `tools/srd_compile` maintainer-tooling precedent, outside `docs_dir` so build internals never ship as site assets) emits one page per public module — page-per-module mirrors the one-home-per-symbol import contract — rendering exactly the `__all__` surface (33 exporting modules, ~560 symbols) through mkdocstrings' Google-style parser; the standing `__all__`-completeness gate keeps the input honest. Cosmetic handler options are implementation's; the rendered-surface pin is not.
- CI, pinned: a `docs` job on ubuntu only (a docs build is platform-independent; the pytest matrix already covers both OSes for the example harness) runs `uv run mkdocs build --strict` on every push and pull request; deployment runs only on push to `main` via the Pages artifact actions (`upload-pages-artifact` + `deploy-pages`), so no `gh-pages` branch ever exists. The one manual step, named for the maintainer: switch the repository's Pages source to "GitHub Actions" once.
- The examples harness, pinned: `pytest-examples` discovers, lints (under the repo ruff configuration, line length 120), and executes every unmarked fenced Python block in `src/` docstrings, `README.md`, and the user-facing site pages. Example house rules, pinned: self-contained, deterministic (fixed seeds — the determinism contract is what makes asserting outputs possible at all), file-touching examples run under a harness-supplied temporary working directory, and assertions target stable facts, never volatile reprs. Two page conventions complete the policy, pinned: on stepwise teaching pages, narrated fragments carry an explicit no-run marker and must be excerpts of the page's complete runnable block, which every teaching page contains — a fragment can typo only if its runnable twin does; and blocks that genuinely cannot run self-contained (client code against a running FastAPI server, interactive TUI transcripts) carry the same marker, rare by design. The marker's concrete spelling follows the harness's native skip mechanism. If `pytest-examples` fights Python 3.14, the fallback is a repo-local collector in `tests/` doing the same discovery and execution, and this plan amends with that decision.

### 2. The docstring overhaul — every public module

- The charter is the audit, its findings re-verified at planning time: Phase 6's stale-promise sweep already removed every "Phase N" reference from `src/`, and TODO/FIXME stands at zero — but "pinned" survives at 259 occurrences, `Examples:` blocks still number exactly two package-wide, no command documents its rejection codes or emitted events, and the rejection vocabulary — 126 distinct codes at planning time, measured by the work item 3 scan (the audit's ~106 was the pre-freeze estimate) — has no reference anywhere. The audit's triage stands: roughly 12–15% of public docstrings need a full rewrite, ~60% a targeted touch-up, ~25% none — and the near-shippable set (`dice.py`, `clock.py`, `abilities.py`, `errors.py`, `data/__init__.py`, `party.py`, the `dungeon.py` geometry orientation) is explicitly protected from churn.
- The audience, pinned: every public docstring addresses a developer who installed from PyPI and cannot see this repository. The decision-log vocabulary ("pinned", "the spec", "census", "seam", "precedent") is purged; repo-relative citations (`docs/adaptations.md`, `srd/*.md`, test names) become plain-language references — "the adaptations register" (a site page after work item 4), "the OSE SRD" — that resolve for a reader without the checkout. Where a purged sentence carried rationale a maintainer needs, the rationale moves to a `#` comment at the enforcement site; design history already lives in the phase plans. Two survivals, pinned: adaptation and pinned-interpretation statements stay in their owning docstrings — the register promises "every entry is also stated in the owning docstring" — reworded for users ("osrlib adopts the reading that…"); and "registered" stays where it is genuine API vocabulary (listener registration), dying only with the decision-log sentences that carried it.
- The quickstart, pinned: one compact cross-seam example — characters → party → adventure → `GameSession` → `execute` → events → `format_message` → save → load — written once and carried verbatim in two places, the `osrlib/__init__.py` package docstring and the README, with a twin-equality test asserting the two blocks identical; the site quickstart page (work item 4) expands the same path stepwise. Entry points it crosses link back to it.
- Entry-point examples, the named set: character creation, `roll`, the attack sequence, casting, treasure generation, `GameSession.execute`, `format_message`, save/load, and `session.view` each gain a runnable `Examples:` block under the harness. One example rule from the audit honored throughout: at least the quickstart crosses module seams; à la carte examples stay local.
- The command contract, pinned layout and gates: every command class docstring gains three sections — `Modes:`, `Rejections:`, `Events:` — and three mechanical gates back them: the modes listed must equal the class's `allowed_modes`; every rejection code named must exist in the source-scanned vocabulary (work item 3); every event named must be a registered event class. The gates kill phantom entries; completeness of the lists is editorial and belongs to review. The rendering, pinned so it reads as chosen: griffe's Google-style parser treats these unrecognized section titles as generic admonition blocks rather than Args-style field lists — they render cleanly, strict builds don't warn, and the gates parse raw docstrings, independent of rendering. The `TOWN → EnterDungeon → EXPLORING` prerequisite the audit called load-bearing-but-undocumented is stated on every command it gates. Event docstrings state when the event fires and enumerate their closed value sets (stances, outcomes); their `allowed_codes` are already declared and template-total.
- The combatant convention, pinned: the duck-typed `Any` parameters stand (the freeze), and the prose does the naming — every such parameter's description names the accepted types with cross-references ("the attacking combatant: a [`Character`][…] or [`MonsterInstance`][…]"), and `combat.py`'s module docstring states the convention once, canonically, for the others to reference. Id-typed parameters (`class_id`, `item_id`, `template_id`, …) cross-reference the loader that enumerates them and the content-id index (work item 3).
- Module docstrings orient: what the module holds, what to call first, where it sits in the core/crawl layering — with import-graph justifications (`alignment.py`'s docstring is currently 100% import-cycle rationale) demoted to comments. Priority runs worst-first per the audit: the `Ruleset` attributes block, then `combat.py`, `spells.py`, `items.py`, `session.py`, `commands.py`.
- The ~30 undocumented private command handlers, resolved: the user-facing contract lands on the public command classes; private handler docstrings are maintainer-facing, welcome where the pass touches them, and not gated by this phase.

### 3. The generated references — `tools/docs/`

- The command/event JSON Schema reference, pinned: generated at build time from the registries themselves — `ALL_COMMAND_CLASSES` (44 commands) and `ALL_EVENT_CLASSES` (68 events: 27 kernel, 41 crawl) — so omission is structurally impossible: per-type pages with each model's `model_json_schema()` output and its modes (read from `allowed_modes`), plus two downloadable raw artifacts, `commands.json` and `events.json`, carrying the discriminated-union schemas. The raw artifacts are the deliverable the spec promises the LLM consumer: schemas an agent framework loads as tool definitions without scraping HTML.
- The rejection-code reference, pinned mechanism: the vocabulary has no runtime registry and the freeze forecloses adding one, so the generator scans `src/` ASTs for `Rejection(code="…")` literals (every current construction site is a literal; if implementation finds a dynamically built code, this plan amends with its handling) and merges one-line descriptions from a hand-maintained `tools/docs/rejection_codes.json` (beside the generators, outside `docs_dir`). Generation fails on a scanned code with no description or a described code no longer in the source — the `srd-data` drift-gate discipline applied to docs. The same scan is the gate behind work item 2's `Rejections:` sections.
- The event message-code table: generated from `allowed_codes` and the template registry (110 codes, already template-total by the standing exhaustive-template test) — code, owning event, default visibility, and the default English template, so front-end authors see what ships before overriding it. The visibility column's source, pinned: the event class's `visibility` field default (there is no per-code static source), and the column says so — specific emissions may override, as the referee `SetDoorState` handler does when it emits the door open/close codes at referee visibility.
- The RNG stream-key reference: a hand-written page for the 12 `*_STREAM` constants (they are stable, few, and need prose about what each stream governs), with a completeness test asserting the page names every `*_STREAM` constant exported from the package.
- The content-id indexes, pinned: generated from the data loaders — class, spell (with level and list), monster, equipment, magic-item, and language ids with display names, and treasure types A–V — identification columns only, never stat-block republication. These pages are SRD-derived: they carry the OGC marking, and the site gains a licensing page with the OGL 1.0a Section 15 notice and the trademark disclaimer, mirroring the README's split.
- Nothing generated is checked in, pinned: reference pages and schema artifacts materialize at build time, so there is no committed output to drift and no regen gate to run — the strict build and the drift tests above are the whole discipline.

### 4. Guides, walk-throughs, and the README — `docs/`, `README.md`

- The nav, pinned: Home; Getting started (Quickstart; Building an adventure); Guides (Sessions, commands, and events; Views and visibility; Determinism, saves, and replay; The kernel à la carte; Listeners and flags; Authoring custom classes and spells; Ruleset options); Front ends (The TUI crawler; The FastAPI pattern; LLM referees); Reference (API; Commands; Events; Rejection codes; Message codes; RNG streams; Content ids; Adaptations register); Licensing.
- Building an adventure kills the audit's construction wall: the worked minimal dungeon — two cells, an edge, an entrance, one keyed area — assembled model by model, validated, entered. The audit's blind test failed here hardest; this page is the fix, and its code blocks run under the harness like every other.
- The guides teach the spec's contracts in prose the docstrings can then stay terse about: the command/event loop and session modes; player/referee visibility and the never-trust-the-client wire discipline; the determinism contract, stamped documents, migrations, and load-equals-replay; the kernel à la carte for the scripts-and-simulations consumer; the listener/flags extension surface retold from the TUI's fetch quest; the authoring guide the Phase 6 groundwork suite deliberately seeded (the definition shape, progression rows, caster profile params, catalog validation — honest that shipped content arrives through the SRD pipeline and that the fixture path is how a game validates its own); and the `Ruleset` flags with the adaptations register linked. The Phase 6 import-homes deferral lands here by name: `osrlib.errors` is taught on the FastAPI pattern page (the status mapping is its consumer) and in the sessions guide (the rejection-versus-exception split), `osrlib.versioning` in the determinism/saves guide (stamps, `SCHEMA_VERSION`, migrations).
- The front-end walk-throughs excerpt the real example sources via snippet markers (marker comments are allowed in `examples/` — they are not the frozen surface; `src/` stays docstring-and-comment-only), so the pages cannot drift from the code they teach. The example READMEs stay the run instructions; the pages teach the patterns — each states its scope so neither forks the other. The LLM-referee page assembles what already ships: the raw schemas as tool definitions, referee visibility and referee commands as the agent's surface, determinism as the replay/eval story — and points at the post-1.0 example-agent deferral rather than promising it.
- The adaptations register joins the nav as-is, with repo-relative links adjusted to render on the site; it stays the single register, now readable by the PyPI user the docstrings will point at it. Its "locked by" test citations stay, pinned: the register is the project's evidence ledger, and stripping provenance to cosmetically match the docstring audience rule would trade information for polish — the page's introduction states that file and test citations refer to the repository.
- The README rewrite, pinned: what osrlib is and the four consumers, honest pre-release status in one line (install from source; PyPI is Phase 8's), the twin quickstart (tested by the harness, equality-gated against the package docstring), the determinism section it already does well, pointers to the site, the licensing split. The phase-ledger development-status section dies — the git history and phase plans are its archive.

### 5. Tests

- **The harness**: every unmarked fenced Python block in `src/` docstrings, `README.md`, and user-facing `docs/` pages lints and runs, on both OSes via the existing pytest matrix; marked no-run blocks follow the work item 1 conventions (excerpt fragments and server-dependent code only).
- **The twins**: the README quickstart block and the `osrlib/__init__.py` quickstart block are asserted byte-identical.
- **The vocabulary tripwire**: a test scans `src/` docstrings for the banned decision-log terms, pinned as a starting list — "pinned", "Phase" followed by a digit, "the spec", "the audit", "census", "seam", "precedent", "rubber-duck", and repo-relative `docs/`, `srd/`, `tests/` path references — case-sensitive phrase matches with an explicit, review-visible allowlist for legitimate uses (prose about a `DungeonSpec` instance must not trip "the spec"). The tripwire backstops the purge; it is not the review.
- **The command contract**: per command class — `Modes:` equals `allowed_modes`, every `Rejections:` code exists in the AST-scanned vocabulary, every `Events:` name is a registered event class.
- **The reference drift gates**: the rejection scan matches the description file's keys exactly; the stream-key page names every `*_STREAM` constant; the docs CI job's strict build covers the rest (cross-references, nav, schema generation) by failing the build.
- **Standing suites**: every existing test passes with zero sanctioned changes — no assertion in the standing suites moves this phase. The golden files are byte-identical and unregenerated, `SCHEMA_VERSION` stays 2 with no new migration, the package version stays 0.1.0, and the pyright, ruff, `__all__`-completeness, and srd-data gates stand as-is. Any golden diff, schema diff, or non-docstring `src/` diff in this phase's changes is a defect by definition.

## Sequencing

1. The toolchain first: the `docs` group, the site skeleton with the generated API reference, the strict-build CI job and Pages deployment, and the examples harness — the checking infrastructure exists before the content it checks, and the strict build turns the docstrings' existing cross-references into checked contracts on day one (dangling ones surface here, fix in step 3's sweep).
2. The generated references next: schemas, rejection codes and their description file, message codes, stream keys, content ids, licensing — they are the link targets the overhaul writes toward, and the rejection scan must exist before the command-contract gate can.
3. The overhaul: the quickstart twins land first as the north star, then the module passes in layer order — `data`, `core`, `crawl`, top-level — worst-first within each layer per the audit triage, with the vocabulary tripwire landing beside the final sweep (it fails until the purge completes).
4. Guides and walk-throughs against the settled docstrings and references.
5. The README rewrite, then the closing pass: every deferral in this plan verified against its pickup (Phase 8 or post-1.0), the docstring audit annotated as executed, the README status line updated, and the site live from `main`.

## Definition of done

- `uv sync && uv run ruff format --check && uv run ruff check && uv run pyright && uv run pytest && uv run mkdocs build --strict` passes locally and in CI on both OSes plus the docs job, with the new gates — the examples harness, the twins, the vocabulary tripwire, the command-contract checks, the reference drift gates — green beside the standing suites, the golden files byte-identical, `SCHEMA_VERSION` still 2, the package version untouched at 0.1.0, and the phase's entire `src/` diff consisting of docstring and comment hunks only.
- The site is live at the pinned URL, deployed from `main`, nav complete: the API reference renders the full `__all__` surface of every exporting module, all 44 command and 68 event schemas ship as pages and as raw `commands.json`/`events.json`, every scanned rejection code carries a description (126 at planning time), the message-code table covers all 110 templated codes, the stream-key page all 12 constants, the content-id indexes carry the OGC marking, and the licensing page carries the Section 15 notice and trademark disclaimer.
- The overhaul holds: zero vocabulary-tripwire hits; every command documents its modes, rejection codes, and emitted events under the mechanical gates; event docstrings state firing conditions and closed value sets; the named entry points carry harness-run examples; module docstrings orient; the combatant convention is stated canonically and referenced from every duck-typed parameter; id parameters link their loaders and indexes; no public docstring cites a repo-internal path as authority.
- The README is rewritten around the tested twin quickstart with an honest one-line status, and the audit's protected near-shippable modules show no gratuitous churn.
- Acceptance, pinned: the implementation-phase rubber-duck includes a fresh-eyes, docs-only walkthrough — the audit's five blind-test tasks plus two new ones (author a custom class from the authoring guide alone; drive a scripted session over HTTP from the FastAPI page alone) — and all seven must be achievable from the published docs without reading library source.
- Every pinned decision in this plan is recorded here and reflected in the shipped configuration; `docs/adaptations.md` gains no entries; and every deferred item names Phase 8 or its post-1.0 track.
