# Phase 5 plan — treasure and the loop closed

Implementation plan for Phase 5 of [the osrlib spec](spec.md). Phase 5 delivers the treasure economy and closes the game loop: the treasure-type tables A–V with hoard generation, gem and jewellery values, the magic item catalog compiled from the generation tables and the per-item pages with structured mechanics where automatable and `manual` tags elsewhere, magic items in play (potions, scrolls, wands and staves, rings, enchanted arms and armour, cursed items, identification), the two remaining `Ruleset` flags (`magic_item_death_save`, `xp_award_timing`), the `TreasureRef` semantics `core/monsters.py` has carried faithfully-but-inert since Phase 2, NPC adventuring parties generated by the SRD procedure and fielded as encounter sides (ending the wandering table's re-roll), the end-of-adventure XP award consuming the defeated-monster ledger Phase 4 accumulated — and replacing the recovered-treasure ledger with an honest brought-back valuation, carrying the first real schema migration — town treasure-selling and paid healing, the dungeon stocking and unguarded-treasure tables as data, and the `examples/` TUI crawler whose fetch quest lives entirely in example-game code on the listener/flags surface. Milestone from the spec: a playable minimal dungeon crawl from character creation to leveling up, including a simple fetch quest implemented in the example's own game code on the listener/flags extension surface — proving games don't need library changes for game-design systems.

## Scope

In scope:

- `tools/srd_compile/treasure.py` — treasure types A–V from `Treasure_Types.md` (bulleted prose, not pipe tables — a bespoke entry parser), gem values and jewellery dice from `Gems_and_Jewellery.md`, the magic item type master table from `Magic_Items_%28General%29.md`, the room-contents and unguarded-treasure tables from `Designing_a_Dungeon.md`; `tools/srd_compile/magic_items.py` — the eight category generation sub-tables, the per-item catalog (84 item pages plus the inline sword/weapon/armour outcome grammar), the sentient-sword tables, and a `_MAGIC_ITEM_MECHANICS` hand-curated map (the `_SPELL_MECHANICS` precedent)
- `src/osrlib/data/` — generated `treasure.json` and `magic_items.json` with typed loaders `load_treasure_tables()` and `load_magic_items()`
- `core/treasure.py` — the new kernel module per the spec architecture: hoard, individual, group, and unguarded-treasure generation, gem/jewellery valuation, magic item rolling with charges, quantities, and sword sentience; à la carte per the spec (`treasure.generate(TreasureType.D, rng)`)
- `core/items.py` — magic item templates and instances (bonuses, charges, identification, curses), valuables (gems, jewellery), treasure weights joining `treasure_weight_coins`; `core/npc.py` — the NPC adventurer party generator (the `validation.py` new-module precedent); `core/combat.py` — enchanted attack/damage/AC wiring and the magic-item death save; `core/ruleset.py` — `magic_item_death_save` and `xp_award_timing`
- `crawl/` — treasure in play (hoard placement, carried-treasure spawns, loot piles), the `UseItem` command and the widened `use_item` battle declaration, ring slots, NPC parties as encounter sides with a casting-capable default policy, the wandering `npc_party` rows going live, the `SpawnNpcParty` referee command, the XP award procedure on town arrival, `SellTreasure` and `PurchaseHealing`
- `osrlib/messages.py`, `crawl/events.py`, `core/events.py`, `crawl/views.py` — the Phase 5 event census, templates, and the player-view masking rules for unidentified items and hidden charges/durations
- `examples/tui_crawler/` — the playable terminal crawler, its authored mini-adventure, the fetch-quest listener, and a scripted mode driving the integration test
- Tests: data pipeline and table fidelity, treasure statistics (the chi-square trio's final third), kernel and crawl suites, the hoard and NPC-party goldens, the phase 5 milestone golden, leak-test extensions

Out of scope (deferred to the phase or track that picks each up):

- Wilderness and the High-Level party types (out of 1.0) — `Adventuring_Parties.md`'s three High-Level compositions and the Expert 75% mounted chance are wilderness-table material and defer with wilderness; the compiler captures the two dungeon kinds only
- Retainers (out of 1.0) — XP divides evenly among surviving party members; the retainer −50% XP penalty and half-share treasure conventions (`Retainers.md`) defer with retainers
- Sentient-sword control in the crawl — generation is complete (intelligence, alignment, powers, ego roll as data) and the control check ships as an à la carte kernel function, but the crawl never auto-invokes ego checks or alignment-touch damage in 1.0; games narrate via referee commands (registered)
- Wishes, magical research, and the referee-judgment item census — `manual`-tagged prose per the Phase 3 spell precedent; the pinned wired census is work item 4
- Identification by hired NPC magic-user (town service) — trial-and-error identification per work item 4 is the 1.0 surface; the weeks-long paid service stays prose (registered)
- Procedural dungeon generation (out of 1.0) — the stocking and unguarded-treasure tables compile as data with à la carte roll functions; no generator consumes them
- Treasure-map hoard placement — rolled maps become inventory items carrying their printed hoard recipe as data; placing the mapped hoard is game content (registered)
- Selling magic items — "no fixed sale value" per `Treasure_Types.md`; `SellTreasure` refuses them (registered)
- NPC parties in keyed-area content — the Phase 5 charter is the encounter tables; keyed content wanting an NPC party uses the new `SpawnNpcParty` referee command from a listener
- Monster spellcasting in the default policy — unchanged from Phase 4; casting tags stay `manual`, and only the new NPC-party policy casts (through the character model, as the spec assigns to Phase 5)

## Work items

### 1. Treasure tables — `tools/srd_compile/treasure.py`, `core/treasure.py` models

- `compile_treasure(srd_dir)` joins `__main__.py` with the established conventions; sources `Treasure_Types.md`, `Gems_and_Jewellery.md`, `Magic_Items_%28General%29.md`, `Designing_a_Dungeon.md`. Output `treasure.json`. The A–V lists are `###`-headed bullet lists, not pipe tables — the parser walks headings and parses each bullet against the pinned entry grammar: optional `NN%: ` gate, then either dice-quantity-with-optional-multiplier plus a glued coin suffix (`1d6 × 1,000cp`, `3d8cp`, `1d4 × 10,000gp` — `×` is U+00D7, thousands commas strip, the multiplier folds into the dice expression's `×K`), a gem/jewellery count (`6d6 gems`, `1d4 × 10 pieces of jewellery`), or a magic-item clause. Magic-item clauses are structured, not free text: `3 magic items` (any), `1 magic sword, suit of armour, or weapon` (a category pool: sword/armour/weapon), `3 magic items (not weapons), plus 1 potion, plus 1 scroll` (an any-roll with a category exclusion plus fixed extras), `2d4 potions` (a diced category count). The compiled `TreasureTypeTable` rows carry `chance_pct` (0 = always), `coins` (denomination + dice), `gems_dice`, `jewellery_dice`, and `magic: tuple[MagicAllotment, ...]` where `MagicAllotment` is `(kind ∈ {any, category, pool}, categories, count | count_dice, exclude)` — `pool` is Type B's "magic sword, suit of armour, or weapon" choice; resolution pins in work item 3. Section membership compiles as `kind ∈ {hoard, individual, group}` (A–O, P–T, U–V) with the average-gp figure parsed from each heading for the statistical tests.
- Gems: the d20 value table (10/50/100/500/1,000 gp bands) verbatim. Jewellery: `3d6 × 100` gp. The damaged-jewellery −50% and the referee combining-values option stay prose (`manual` notes on the table model) — nothing in the engine crushes jewellery in 1.0, and combining is presentation (registered).
- The magic item type master table compiles with both printed d% columns as `basic` and `expert` band sets (`01–10 … 91–00`; `00` reads as 100). Tier semantics pin in work item 3.
- `Designing_a_Dungeon.md` contributes two tables: the room-contents d6 (`1–2 Empty … 6 Trap`) with its per-contents treasure chance column (`1-in-6`, `3-in-6`, `None`, `2-in-6` — ASCII-hyphen values beside an en-dash d6 column, per the survey), and the unguarded-treasure bands (Level 1, 2–3, 4–5, 6–7, 8–9) parsed from their `**Level N:**` prose lines with the same entry grammar as the treasure types — note the sp entries are ungated everywhere, the gp entries are 50%-gated on the first two bands only, and nothing about the band schema is uniform. Both ship as data with à la carte rolls (work item 3); no crawl procedure consumes them in 1.0.
- Models (`TreasureTypeTable`, `GemValueTable`, `MagicItemTypeTable`, `StockingTable`, `UnguardedTreasureTable`) live in `core/treasure.py` — the `core/tables.py` layering precedent: loaders import their model homes, `core` imports loaders, `crawl` consumes.

### 2. The magic item catalog — `tools/srd_compile/magic_items.py`

- `compile_magic_items(srd_dir)` produces `magic_items.json`: every generation sub-table plus a per-item template catalog. Sources: `Potions.md`, `Scrolls_and_Maps.md`, `Rings.md`, `Rods%2C_Staves%2C_Wands.md`, `Miscellaneous_Items.md`, `Swords.md`, `Weapons.md` (the magic weapons page — the mundane page is `Weapons_and_Armour.md`), `Armour_and_Shields.md`, the four sentient-sword pages, and the 84 per-item pages.
- Sub-table conventions, pinned from the survey: each category table carries a sparse small-die B column (blank cells are not data — B and X are independent index spaces over the same outcome list) and a full d% X column; `00` = 100; `Miscellaneous_Items.md` alone uses ASCII hyphens in its ranges (every other table uses en-dashes) and contains the degenerate `94-94` — the range parser accepts both dash forms and single-value ranges. The scroll spell-level table's two-row spanning header (`|  |  | Spell Level | |` above the real header) gets the encounter-table special-case treatment. Rows whose printed links have no backing page (the six `Potion_of_Control_*`, the two `Crystal_Ball_with_*`, `Ring_of_Protection,_5'_Radius`, the four elemental devices) resolve to hand-mapped ids against the combined pages — the `_TABLE_ENTRIES` precedent; the compiler asserts every sub-table outcome resolves to a catalog id. Catalog ids are per printed variant — the six control potions, the three crystal-ball forms, the four elemental devices, and both ring-of-protection forms each get their own resolvable id (the `oil_beetle` per-variant precedent) — while payload-only bands (the three ring wish-count rows) compile as one id whose count dice are generation params, pinned. `Armour_and_Shields.md` additionally carries the *Magic Armour Type* d8 table (1–2 leather, 3–6 chainmail, 7–8 plate mail), compiled alongside its outcome table — it is what gives a generated `Armour +2` its `base_item_id` (work item 3's roll order).
- `MagicItemTemplate` (frozen, in `core/items.py`): `id`, `name`, `category ∈ {armour, misc, potion, ring, rod, staff, wand, scroll, sword, weapon}`, `base_item_id` (the mundane `equipment.json` template an enchanted arm overlays — sword, dagger, chainmail, shield, arrows …), `attack_bonus`/`damage_bonus`/`ac_bonus` (negative for cursed), `versus: tuple[VersusBonus, ...]` (the `+2 vs Lycanthropes` clauses as structured category/monster-tag references with their alternate bonus — the `_MAGIC_ITEM_MECHANICS` map enumerates every printed versus-target's tag mapping, from lycanthropes and undead as category tags through spell users and regenerating creatures as ability-derived template sets, so no clause resolves by string-matching prose), `cursed: bool`, `charges_dice` (`1d10` rods, `3d10` staves, `2d10` wands, per-item overrides; `None` for uncharged), `quantity_dice` (arrows/bolts; the tier-conditional `Basic: 10; Expert: 2d6` cell compiles as both fields), `usable_by` (structured: class-id set or caster-kind — wands arcane, staves per page, scrolls per work item 5), `always_active: bool` (swords, weapons, armour, protective items per `Magic_Items_%28General%29.md`), `effect: EffectSpec | None` (structured mechanics for the wired census — modifier kinds, damage dice, save category, area shape, duration), `params`, `manual: tuple[str, ...]` (tagged prose for everything unwired), `weight_coins` (the base item's weight; enchanted armour at half per RAW; potions/scrolls from the `TreasureWeight` rows).
- Swords/weapons/armour have no per-item pages — their outcomes compile from the inline table-cell grammar (`Sword +1, +3 vs Dragons`, `Cursed Armour -2 with Shield +1`, `Dagger +2, +3 vs orcs, goblins, and kobolds`): leading bonus, optional versus clause, optional paired second item (the armour-with-shield rows compile as two-item bundles). Comma-tokenizing is banned; the grammar is positional.
- Sentient swords: the communication/languages/alignment/powers tables and both power tables (with their `Roll twice again` composite rows and duplicates-allowed annotations) compile as data; `SwordSentience` generation params ride the sword rows — the special-purpose 1-in-20 rolls first per magic sword (a special sword is always sentient at INT 12/Ego 12, RAW), otherwise the 30% sentience roll, then the page's remaining steps in its printed procedure order (INT 1d6+6 through Ego 1d12), pinned. Powers are `manual`-tagged data (sensory and extraordinary alike) — the control-check procedure is work item 4's kernel function; power automation is out of 1.0.
- `Helm_of_Telepathy.md` is truncated mid-sentence in the SRD dump — its description completes via `overrides/magic_items.json` with provenance, the "Basic Adventures" precedent.
- Scroll sub-structures: spell-count outcomes (1/2/3/5/7 spells), the 1-in-4 divine gate, the random spell-level table (arcane and divine columns), the four protection scrolls as structured params — lycanthropes and undead carry per-HD-band affected-count dice with 6-turn durations, elementals bars all elementals for 2 turns, magic runs 1d4 turns — the cursed scroll with its six example curses as data rows (transformation, summoning, lost item, energy drain, ability-score re-roll, slow healing), and treasure maps I–XII compiled as hoard recipes (reusing the treasure-type entry model).

### 3. Kernel treasure generation — `core/treasure.py`

- `TREASURE_STREAM = "treasure"` joins the stream census; every draw in this work item belongs to it. Generation order within any one table is pinned: entries resolve in printed order — presence roll, then quantity dice, then per-item resolution (each gem's value, each jewellery's value, each magic item fully depth-first: type roll if unspecified, sub-table roll, charges/quantity/sentience) — so results are reproducible from the stream alone.
- `generate_treasure(treasure_type, *, tier, stream, allocator) -> GeneratedTreasure`: coins (a `Coins`-shaped bundle), `valuables: tuple[ValuableInstance, ...]`, `magic_items: tuple[MagicItemInstance, ...]` — instance ids come from the caller's `IdAllocator` (the session passes its own; à la carte callers construct one), prefixes `valuable` and `magic-item`. `generate_unguarded_treasure(dungeon_level, *, tier, stream, allocator)` rolls the level band (clamped into the printed bands, the encounter-table precedent; levels 10+ use the 8–9 band). `roll_room_contents(stream)` exposes the stocking d6 + treasure-chance pair à la carte.
- `generate_magic_item(category_or_any, *, tier, stream, allocator, exclude=())` rolls the master table when unspecified (re-rolling excluded categories — draws consumed, the wandering-re-roll precedent), then the category sub-table, then instance details: the armour-type d8 for armour outcomes (setting `base_item_id` to leather, chainmail, or plate), charges (rolled at creation, referee-only forever after — RAW "undiscoverable"), arrow/bolt quantities, paired armour bundles, wish counts, and sword sentience in work item 2's pinned order (special purpose first, then the 30% roll, then the printed procedure). Allotment semantics, pinned: a `pool` allotment picks uniformly among its categories (treasure stream) before the sub-table roll; an exclusion (`not weapons`, map IV's `no swords`) excludes exactly the named master-table category — swords and weapons are distinct master-table types and exclude independently.
- Tier, pinned: `tier ∈ {basic, expert}` selects the printed B or X probability columns; the crawl passes `basic` while the party's highest living level is 1–3 and `expert` at 4+ (the table's own definition of Basic and Expert characters), evaluated at generation time. À la carte callers choose explicitly. The SRD's "always use Expert for even distribution" alternative is a caller choice, not a flag — the `Ruleset` register stays the spec's closed 1.0 set.
- Gems roll the d20 value table per stone; jewellery rolls `3d6 × 100` gp per piece. `ValuableInstance` (mutable, serialized): `instance_id` (allocator prefix `valuable`), `kind ∈ {gem, jewellery}`, `value_gp`, `weight_coins` from the `TreasureWeight` rows. Values are rolled at generation and fixed thereafter — appraisal is instantaneous and exact, pinned (B/X prices treasure for the XP economy; a haggling/appraisal minigame is game territory, registered).
- The `TreasureRef` semantics, pinned at last (closing the Phase 2 docstring promise): `letters` resolve each by its own section — hoard letters (A–O) are lair treasure, individual letters (P–T) generate once per monster, group letters (U–V) once per group; `parenthetical` letters are lair treasure regardless of section (the Bandit's `U (A)`: U carried by the group, A in the lair); `extra_gp` adds flat gp to the lair hoard; `multiplier` repeats the whole listed generation that many times (the Noble's `V × 3`); `special` labels (Tusks, Honey, Horn) and `see_below` generate nothing — they are content prose for keyed areas (registered). The small-lair reduction and the referee's manual value adjustment stay prose (registered — both are explicit referee discretion).

### 4. Magic items in the kernel — `core/items.py`, `core/combat.py`, `core/effects.py`, `core/ruleset.py`

- `MagicItemInstance` (mutable, serialized): `instance_id` (prefix `magic-item`), `template_id`, `charges_remaining` (referee-only), `quantity` (arrows), `identified: bool = False`, `sentience: SwordSentience | None`, `state: dict` (per-item memory: staff-of-healing per-target days, invulnerability last-use day). Inventory holds them beside `ItemInstance` (an additive union member); `EquipItem`/`UnequipItem` accept them wherever their `base_item_id` kind fits, plus the new ring slots: `Inventory.rings` (max two — a third rejects `items.ring.hands_full`; RAW, more than two = none function, delivered as the slot cap).
- Enchanted arms in combat: an equipped magic weapon contributes `attack_bonus`/`damage_bonus` (versus-clauses swapping in their alternate bonus when the target's template carries the referenced tag or id), counts as magical for the graded-immunity checks (the Phase 2 seam — `counts_as_magical` now has a real source besides the `hd5_counts_as_magical` flag), and cursed negatives apply identically. Magic armour and shields apply `ac_bonus` to the wearer's AC; the cursed `AC 9 [10]` form sets AC outright (`ac_set`). Item bonuses are computed from equipped inventory at query time — never `ActiveEffect` stat modifiers — which is exactly the `modifier_total` carve-out `core/effects.py:329` reserved: item bonuses combine freely with spell modifiers and are exempt from the cumulative caps. Item-sourced ledger effects (potions, wards) attach with kind `item` and `dispellable=False`, closing the `core/spells.py` dispel-exemption seam; the cumulative largest-bonus cap in `modifier_total` applies among spell-kind effects only — `item`-kind effects are exempt, pinned (RAW's carve-out covers magic items generally, not just worn ones).
- The wired census, pinned (everything else in the catalog is `manual`-tagged prose, the Phase 3 spell-census precedent): all ±N swords, weapons, arrows/bolts, armour, and shields including versus-clauses and cursed forms; Sword +1 Energy Drain (on-hit `drain_levels` with the `level_minimum` XP policy — the sword's page sets XP to the lowest amount for the new level, unlike the wight's halfway — draining automatically on every hit, pinned CRPG default over RAW's "may command" and registered; 1d4+4 total drains then a plain +1, tracked in `state`) and Sword +1, Light (the Phase 3 light-effect machinery); potions of fire resistance, giant strength (damage ×2), growth (melee damage ×2), healing (1d6+1 or cure paralysis), invisibility (the Phase 3 condition, broken on attack), invulnerability (+2 AC and saves, with the more-than-once-per-week inversion to −2 tracked per character on the clock's day), poison (save vs poison or death), and speed (movement ×2, attacks ×2 — the *haste* machinery); rings of protection (+1 AC and saves, always active; the 5'-radius variant extends to the wearer's rank, pinned — the battle space has no finer adjacency), fire resistance, regeneration (the troll-regeneration machinery with its `while_alive` gate — the ring does not function at 0 hp or below, RAW, unlike the reviving troll; 1 hp/round, fire and acid damage unregenerable), and weakness (cursed: a 6-round onset then `strength_set` 3 — `strength_set` joins `MODIFIER_KINDS` beside `ac_set`, additive, and serves Gauntlets of Ogre Power too); wands of cold, fear (save vs wands or the `afraid` rout), fire balls, lightning bolts, and paralysation (6 turns, save) — each 6d6 or per-page, resolving through the kernel damage/save path and the Phase 4 area-footprint rule with the item's printed shape; staff of healing (1d6+1, once per target per day, no charges) and staff of striking (2d6 per charge); protection scrolls against lycanthropes and undead (timed ward effects with their per-HD-band counts as structured params, the *protection from evil* melee-ban machinery) and elementals (all elementals, 2 turns; the magic variant stays `manual`); spell scrolls (work item 5); Gauntlets of Ogre Power (`strength_set` 18, +3 attack/damage unarmed-or-melee per page, +1,000 coins carry), Girdle of Giant Strength (attacks on the 8-HD matrix row; damage is twice normal weapon damage under `variable_weapon_damage` — the default — and the printed 2d8 with the flag off: the one wired item whose mechanics branch on a `Ruleset` flag, named as such in the tests), Bag of Holding (10,000-coin capacity at 600 coins carried weight), and Displacer Cloak (+2 on the listed save categories, melee attackers −2 — missiles unaffected, RAW).
- Identification, pinned as the register's headline adaptation for this phase: generated magic items enter play unidentified; the player view masks an unidentified item behind its category display name ("a potion", "a scroll", "a sword with a faint aura" — the aura concession because `detect magic` exists) and hides bonuses, charges, names, and sentience; first meaningful use identifies — drinking a potion (a sip is the same event at this simulation floor, registered), reading a scroll, activating a device, or the first attack roll/damage taken with an enchanted arm or armour equipped in battle. Cursed items identify as their true nature at the same trigger — the armour-reads-as-+1 deception collapses at first battle use, which is RAW's own reveal moment ("until used in deadly combat"), and a revealed cursed item pins to its bearer: cursed swords, weapons, armour, shields, and rings all reject `UnequipItem`/`DropItems`/`SellTreasure` with `items.curse.stuck` (each category's page carries the same cannot-discard clause) until *remove curse* (town service or the spell). `IdentifyItem` ships as a referee command for game-driven identification.
- The magic-item death save: `magic_item_death_save: bool = True` joins `Ruleset` (the spec's default-on flag). `destroy_equipment` grows a per-item pre-phase: when the flag is on, each magic item in the doomed inventory rolls the owner's save (the destructive source's save category, pinned) plus the item's best combat bonus; survivors land in a drop pile at the victim's cell instead of vanishing (pinned — surviving the blast but not the looting would be no survival at all). `EquipmentDestroyedEvent` grows an additive `saved_items` field.
- `sword_control_check(character, sword, *, stream) -> SwordControlResult` ships à la carte with the RAW arithmetic (sword Will = INT + Ego + 1 per extraordinary power + 1d10 when alignments differ; wielder Will = STR + WIS − 1d4 below full hp, − 2d4 below half), no events, crawl-neutral — the registered 1.0 boundary for sentience.

### 5. Magic items in play — `crawl/commands.py`, `crawl/exploration.py`, `crawl/battle.py`

- `UseItem(character_id, item_id, target_id=None)` — exploring and encounter modes, one round (the RAW activation cost; drinking is one round). The battle-round `use_item` declaration widens from thrown-splash-only to potions (self), wands/staves (target group, resolved in the magic phase alongside casts — pinned: devices unleash magical effects; RAW gives no phase, and magic-phase keeps the missile/melee ordering clean, registered), and scroll reading (magic phase, same pin). Concentration items occupy the round exactly as RAW says — using one is the character's whole action, which the declaration model already enforces; once-per-round is the declaration cadence itself.
- Potions: drinking attaches the item-kind effect with `duration_dice="1d6+6"` turns rolled on the effects stream — referee-visible only; the player view shows potion effects without remaining duration, pinned (RAW: "not tell the player how long the potion will last"). Mixing, pinned as both printed consequences (the SRD lists them as one outcome, not alternatives): drinking while under another potion's effect cancels both effects and disables the drinker for 3 turns (`helpless`-family condition), inapplicable to instantaneous or permanent potions (healing, poison, longevity).
- Spell scrolls: reading casts the inscribed spell through a `cast_from_scroll` kernel entry that reuses `validate_cast`'s legality gates and the full mode-resolution/effects machinery but skips the memorized-copy consume (the scroll is the copy — it burns instead: "when a scroll is read, the words disappear"). Arcane scrolls require an arcane caster — or a 10th+ level thief through the class's `scroll_use` ability, whose printed 10% chance of error (magic stream) resolves as a simple fizzle consuming the spell, pinned and registered (RAW's "unusual or deleterious effect" is referee territory); divine scrolls require a divine caster; reading requires light (the Phase 4 light gates). RAW's arcane magical script ("can only be read by magic" — the rule the Phase 3 plan assigned to scroll items) is waived at this simulation floor, pinned and registered: reading-to-cast needs no prior *read magic*, the identification model of work item 4 being the gate that matters. Scroll spells resolve at the minimum class level that can cast the spell per the compiled progression, pinned (RAW is silent on scroll caster level; the floor is the least-power reading, registered). Each inscribed spell is single-use; multi-spell scrolls track remaining spells in `state`. Cursed scrolls trigger on the read attempt regardless of class: the curse rolls uniformly from the six compiled examples (effects stream) — energy drain resolves through `drain_levels` and slow healing through the Phase 2 slowed-healing hooks, while transformation, summoning, lost item, and ability-score re-roll are `manual`-tagged in the emitted event — pinned.
- Devices (wands, staves, rods): activation checks `usable_by`, spends one charge, resolves the wired effect against the Phase 4 combat space (cones and spheres through the footprint rule; exploration-mode targets are the encounter's groups, battle-mode the declaration's target group). The last charge exhausts the item to inert — the exhausting use itself resolves normally and nothing announces it (charges are undiscoverable, RAW); the item remains in inventory and activation thereafter rejects with the `items.device.inert` rejection code (a rejection, not an event — the Phase 4 blocked-move precedent), and no recharging exists, RAW. Charges never appear in any player-facing surface.
- Rings: continuously active while worn (equip is the zero-time exploration action, the Phase 4 equip precedent); the two-ring slot cap from work item 4.

### 6. Treasure in play — `crawl/exploration.py`, `crawl/encounter.py`, `crawl/dungeon.py`

- Generation timing, pinned: lazily, at the moment the treasure first becomes observable, all on the treasure stream. A keyed area's lair hoard (the keyed monsters' hoard letters, parenthetical letters, and `extra_gp`) generates when the area's keyed encounter first spawns; the hoard lands as an engine-created cache in `DungeonState.generated_caches` (id from the allocator, prefix `cache`, contents from `GeneratedTreasure`) and emits `treasure.hoard.generated` (referee). Authored `FeatureSpec`s are frozen content and stay that way — the state overlay owns play-created treasure, the Phase 4 template/instance split applied to loot; `InspectTreasure` and `TakeTreasure` resolve authored caches and generated ones through a single lookup, and `DropPile` grows additive `valuables` and `magic_items` fields so drops and loot round-trip. The authoring surface for named treasure: `FeatureSpec` grows an additive `valuables: tuple[ValuableSpec, ...]` (kind, value, display name) instantiated on take — the example's MacGuffin is one. Carried treasure (individual P–T per monster, group U–V per group) generates at spawn for keyed and wandering monsters alike. NPC-party treasure is work item 7. An `AreaSpec` may also declare `treasure: AreaTreasureSpec` — explicit type letters or `unguarded: true` (rolling the level's band) — generating on first entry into the area; this is how content places generated treasure without monsters, and the milestone dungeon uses it.
- Loot flow, pinned: at battle end, slain and surrendered monsters' carried treasure drops as a pile at the party's cell (surrender hands it over; the pile mechanism is already the recovery surface); routed monsters flee with theirs. `TakeTreasure` moves valuables and magic items into inventory exactly as it moves coins and items today. `RecoveredTreasureRecord` and `session.recovered_treasure` are cut outright: their one promised consumer was the award, the award's input is now work item 9's valuation delta, and the event log already records every acquisition — a ledger kept "as a log" with no consumer is exactly the accommodation AGENTS.md bans, and this plan supersedes the Phase 4 comments naming it the award input. Removing serialized session state is a schema change: `SCHEMA_VERSION` bumps to 2, and the 1 → 2 migration (drop the field) is the migration framework's first real exercise, replacing the synthetic-only coverage. `DefeatedMonsterRecord` stays — the award consumes it.
- Valuables and magic items weigh, with the treasure/equipment split pinned: `treasure_weight_coins` grows from purse-only to purse + valuables + the magic-item categories the `TreasureWeight` rows price (potion, scroll, rod, staff, wand), closing the Phase 1 seam its docstring names; rings and miscellaneous items weigh zero absent a page figure (Bag of Holding's printed 600-coin loaded weight rides its params); enchanted weapons and armour weigh as equipment beside their mundane bases — base weight, armour halved per RAW — not as treasure, so basic encumbrance's treasure tracking stays honest. `DropItems` accepts valuables and magic items; dropped cursed-equipped items reject per work item 4.
- `InspectTreasure`/`RemoveTreasureTrap` operate on engine-created caches identically to authored ones (generated hoards are untrapped — traps are authored content, per the stocking table's own separation, pinned).

### 7. NPC adventuring parties — `core/npc.py`

- `NPC_PARTY_STREAM = "npc_party"` joins the census: composition, class/level, alignment, ability scores, hp, and spell picks draw from it; the party's treasure and magic items draw from the treasure stream (they are treasure procedures and belong to its statistics), pinned draw ownership.
- `generate_npc_party(kind ∈ {basic, expert}, *, count, catalogs, npc_stream, treasure_stream) -> NpcParty`: for each member — d8 on the class/level table (rows keyed by die result; results 4 and 5 are both Fighter with different Expert level dice, per the survey), level dice by kind (Basic all 1d3; Expert per row — every printed range respects the demi-human caps, asserted at compile time), 3d6-in-order ability scores through the existing creation kernel, hp by rolling the class hit die per level through `level_up` (CON applied, minimum 1 per level), `xp` set to the class's threshold for the rolled level. Alignment, pinned: one d6 roll on the alignment table for the whole party (RAW offers either; a single alignment drives reaction, parley, and ward interactions coherently, registered).
- Equipment, pinned (RAW says only "normal adventuring gear" — the kit table is invented and registered): Cleric mace + chainmail + shield; Dwarf battle axe + chainmail + shield; Elf sword + longbow with one arrow lot + chainmail; Fighter sword + chainmail + shield; Halfling sword + sling with one stone lot + leather + shield; Magic-User dagger; Thief sword + leather + thieves' tools. Expert-tier Clerics, Dwarves, and Fighters upgrade body armour to plate mail. Every member carries a standard-rations lot, a waterskin, and a torch lot (the gear the survival procedures read).
- Casters memorize by rolling each open slot uniformly from the class-legal spells of that level (npc stream), normal forms (arcane spell books equal exactly the memorized picks), pinned — RAW says "choose or roll" and rolling is the deterministic branch.
- Expert magic items, RAW: per member, per suitable sub-table, a 5%-per-level chance (treasure stream, sub-tables in the catalog's printed order); unusable rolls are ignored, no re-roll. Suitability pins by usability: armour and swords/weapons for classes that may wear/wield them, potions/rings/miscellaneous for everyone, scrolls and rods/staves/wands for casters (arcane restrictions honored per item). Rolled wearable/wieldable items are equipped when better than the kit piece (higher effective AC / any enchantment over a mundane arm), pinned.
- Group treasure: types U + V rolled once, shared — carried as a group bundle that drops with work item 6's loot flow (slain or surrendered; a routed party keeps it).
- Defeat XP, pinned: an NPC adventurer is worth the `Awarding_XP.md` table value for HD equal to their level, no plus-category, no ability bonuses (registered — RAW prices monsters, not classed NPCs; level-as-HD is the straight reading). Their `DefeatedMonsterRecord` uses `template_id="npc:<class_id>"`.

### 8. NPC parties as encounter sides — `crawl/encounter.py`, `crawl/battle.py`, `crawl/exploration.py`, `crawl/session.py`

- The side model generalizes: NPC members register in `session.npcs: dict[str, Character]` (allocator prefix `npc`), and `EncounterGroup.monster_ids` becomes the combatant-id list it always structurally was, resolved through a `session.combatant(id)` lookup spanning monsters and NPCs. The battle machine's attack, damage, save, condition, and effects paths already speak Character (the party is one); the monster-only assumptions to lift are target pools, morale scores, and the action policy.
- The wandering re-roll ends: an `npc_party` row rolls its printed count dice (wandering stream, the Phase 4 count precedent), generates the party per work item 7, spawns it as one encounter group, and the encounter procedure runs unchanged — surprise both ways, 2d6 × 10' distance, reaction (NPC parties react on the table like anyone; parley works; keyed stances don't apply). Evasion and pursuit treat an NPC group as intelligent for the distraction roll regardless of treasure letters, pinned (they are people). `SpawnNpcParty(kind, count_dice=None)` joins the referee commands for keyed content, quest listeners, and tests.
- NPC morale, pinned: NPC adventurer groups check morale at ML 9, the Veteran stat block's printed score — the SRD's own low-level-adventurer monster as the data anchor rather than an invented number (registered; keyed content and games override via policy substitution or referee commands).
- `NpcPartyPolicy` (default for NPC sides, monster-action stream, the `ActionPolicy` protocol): per living member each round, in a pinned priority — (1) a caster holding a memorized cure spell heals the group's most-wounded member below half hp (lowest hp ratio, ties by id); (2) a caster holding a memorized wired offensive spell casts it — highest spell level first, ties by spell id — at the party (areas through the footprint rule, single-target picks uniform from the reachable rank); (3) a member with a missile weapon and a gap beyond 5' shoots; (4) otherwise close and melee, the monster default. NPC casts post declarations and are disruptable exactly like party casts (RAW's trigger doesn't care which side declares), pinned; NPC members drink a carried healing potion when below half hp and no cure is available, pinned as priority (1½) — the one item-use the default policy performs. All of this is registered as invented tactics over RAW's silence; games and LLM referees substitute policies per side as ever.
- Defeated NPCs post `MonsterDefeatedEvent`s (work item 7's records) and the group's carried treasure, kits, and magic items drop per work item 6's loot flow — lootable arms are ordinary equipment templates and magic-item instances, so victory over an Expert party is the campaign's magic-item faucet, exactly as B/X plays.

### 9. The XP award — `crawl/session.py`, `crawl/exploration.py`, `core/ruleset.py`

- `xp_award_timing ∈ {on_return, immediate}` (default `on_return`) joins `Ruleset` — the spec's flag, RAW versus the continuous-CRPG adaptation.
- Recovered-treasure valuation, pinned: the award values what the party actually brought back — at each departure from town (`EnterDungeon`) the session snapshots the party's treasure valuation (all members including the dead: coin value in cp plus every `ValuableInstance.value_gp`; magic items and mundane equipment count zero — magical treasure grants no XP per RAW, and mundane-gear salvage is below the simulation floor, registered); on return the treasure XP is the valuation delta, floored to gp once from the cp total, never negative (clamped at zero — a party that lost money learned nothing monetarily, pinned). The delta handles dropped-and-abandoned treasure, re-taken piles, and quest-listener `GrantCoins` rewards without any acquisition bookkeeping — which is why the recovered-treasure ledger is cut rather than kept (work item 6).
- The award, `on_return`: when `TravelToTown` lands the party in town, after the arrival events the session totals defeated-monster XP (the `DefeatedMonsterRecord` ledger) plus the treasure delta, divides evenly among living members (floor division; the remainder is dropped, pinned — RAW divides evenly and B/X arithmetic is integer), and applies each share through `apply_xp` (prime-requisite modifier and the one-level clamp are already the kernel's), emitting `session.xp.adventure_award` (player — the totals and the per-head share) then one `session.xp.awarded` per member in marching order. The defeated-monsters ledger clears; the next `EnterDungeon` snapshots anew. A TPK never awards (no one returned, RAW). Dead members' carried treasure that made it back counts toward the pool (the party recovered it); dead members receive no share. The Phase 4 note that "Phase 5's award will drive" `AwardXP` is superseded explicitly: the award applies through `apply_xp` directly (a command whose handler executed further commands would double-log), and `AwardXP` remains the referee and game surface — the example's quest bonus drives it (work item 12).
- `immediate`: monster XP divides and applies at each encounter end, treasure XP at each `TakeTreasure` (the acquisition's value as the delta), same division and events; town arrival awards nothing and drops never refund (registered — the arcade adaptation trades exactness for immediacy). `AwardXP` stays the referee surface it was, unchanged in either mode.

### 10. Town services — `crawl/commands.py`, `crawl/exploration.py`

- `SellTreasure(item_ids)` — town mode, zero time: valuables convert to coins at full `value_gp` (pinned and registered: the SRD prices treasure but names no exchange spread; full value keeps the 1-gp-1-XP identity clean). Magic items reject `town.sell.no_fixed_value` (RAW's own words). Selling after the award changes nothing — the value was already counted at return, and the next departure's snapshot absorbs the new coins.
- `PurchaseHealing(character_id, service)` — town mode, zero time; the temple's service list and prices are invented and registered (the SRD's base-town page is prose): *cure light wounds* 25 gp, *cure serious wounds* 100 gp, *cure disease* 150 gp, *neutralize poison* 150 gp, *remove curse* 200 gp, *raise dead* 1,500 gp. Each resolves through the kernel spell path with an abstract temple cleric at the minimum level able to cast the spell per the compiled progression (magic stream for the rolls, effects stream for attachments — the streams the spells already own); *raise dead* and *neutralize poison* read the session's death records exactly as Phase 4 wired for `CastSpell`, so the temple honors the RAW windows. All services are always available in the 1.0 marker town, registered (town size and cleric availability are game territory).

### 11. Events, messages, views — `crawl/events.py`, `core/events.py`, `osrlib/messages.py`, `crawl/views.py`

- The census (event — codes — visibility): `HoardGeneratedEvent` (`treasure.hoard.generated`, **referee** — contents itemized; players learn by finding), `ItemUsedEvent` (`items.potion.drunk` / `items.potion.mixed` / `items.scroll.read` / `items.scroll.cursed` / `items.device.activated`, player — `items.device.inert` is a rejection code, not an event, per work item 5), `ItemIdentifiedEvent` (`items.item.identified`, player), `CurseRevealedEvent` (`items.curse.revealed`, player), `NpcPartySpawnedEvent` (`encounter.npc_party.spawned`, **referee** — full roster; the player-facing `EncounterStartedEvent` names "adventurers" and the count), `AdventureXpAwardEvent` (`session.xp.adventure_award`, player), `TreasureSoldEvent` (`town.treasure.sold`, player), `HealingPurchasedEvent` (`town.healing.purchased`, player). - The player view whitelist extends: inventories show magic items masked-until-identified (category display name; no bonuses, charges, sentience, or true id), valuables with their values (appraisal is exact, work item 3), potion effects without remaining duration, and never a charge count, a hoard not yet found, or an NPC roster beyond what `EncounterStartedEvent` showed.
